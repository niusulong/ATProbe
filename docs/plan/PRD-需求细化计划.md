# 串口AT命令自动化测试工具 — 需求细化计划

> 版本：v1.0
> 日期：2026-05-19

---

## 1. 目标

将总体需求文档按功能模块拆分，逐个深入细化，输出独立的模块需求文档。每份文档聚焦单一功能域，描述**做什么**，不涉及**怎么做**（技术选型、架构设计等留在设计阶段）。

---

## 2. 模块清单与细化顺序

按依赖关系从底层到上层排列，共6个模块：

| 序号 | 模块名称 | 涵盖范围 | 依赖 |
|------|---------|---------|------|
| M1 | 串口通信管理 | 串口配置与连接、多串口并行、URC处理、串口原始日志记录 | 无（基础层） |
| M2 | 测试用例定义 | YAML用例格式、用例组织（套件/标签）、变量系统、前置条件 | 无（定义层） |
| M3 | 测试执行引擎 | 执行控制、三种测试场景（基础验证/命令序列/性能压测）、失败处理、测试上下文 | M1, M2 |
| M4 | 测试报告 | 报告内容、报告格式（HTML/控制台/JUnit XML）、报告管理 | M3 |
| M5 | CLI界面 | 命令与子命令、参数配置、配置文件、进度显示 | M3, M4 |
| M6 | UI管理界面 | ~~Web界面~~（已调整为桌面端 GUI，见 PRD §8.2 / REQ-M6）用例管理、实时执行监控、报告查看 | M3, M4 |

---

## 3. 细化流程

每个模块按以下流程执行：

```
讨论需求细节 → 输出需求草稿 → 用户审阅确认 → 定稿存档
```

**讨论阶段：**
- 针对当前模块逐项讨论功能点、输入输出、边界条件、异常场景
- 一次聚焦一个子话题，避免信息过载

**输出阶段：**
- 生成该模块的详细需求文档，存入 `docs/requirements/` 目录
- 文件命名：`REQ-M<序号>-<模块名>.md`

**审阅阶段：**
- 用户审阅草稿，提出修改意见
- 修改后确认定稿

## 5. 各模块细化要点

以下列出每个模块需要深入讨论的关键点，作为讨论时的检查清单，实际需求以讨论结果为准。

### M1 串口通信管理

- 串口参数配置项的完整列表和默认值
- 串口连接生命周期（打开、关闭、重连策略）
- 热插拔检测机制和自动重连的行为细节
- 多串口并行的资源管理和隔离策略
- URC规则定义方式、匹配模式、触发动作的完整列表
- URC与命令响应的区分机制
- 原始日志的记录时机、格式细节、文件管理策略
- 串口异常场景的处理（断连、超时、数据错乱）

### M2 测试用例定义

- YAML用例的完整结构（单命令用例、命令序列用例、压测用例）
- 断言类型的完整列表和语义
- 变量提取和引用的规则
- 条件分支和流程控制的表达方式
- 用例套件的组织结构和引用关系
- 标签系统的设计（预定义标签、自定义标签）
- 前置条件检查的定义方式
- 用例间变量传递机制

### M3 测试执行引擎

- 三种测试场景的执行流程细节
- 单命令基础验证的完整流程
- 命令序列执行的步骤间交互机制
- 性能压测的循环控制、并发控制、统计维度
- 命令序列压测的双维度统计（单步 + 整体）
- 失败处理策略的可配置方式和优先级
- 超时控制的三级机制（单步、用例、全局）
- 测试上下文的变量生命周期和作用域

### M4 测试报告

- 报告概览的统计项和展示方式
- 用例详情的展示粒度
- 失败详情的信息完整性
- 性能压测报告的统计项和图表需求
- URC日志在报告中的呈现方式
- HTML报告的样式和交互需求
- 控制台输出的实时展示格式
- 报告的存储和命名规则

### M5 CLI界面

- 子命令的完整列表和职责
- 每个子命令的参数列表和默认值
- 配置文件的格式和支持的配置项
- 执行进度的展示格式
- 错误提示和帮助信息的设计
- 多串口并行执行时的CLI输出组织

### M6 UI管理界面

- 页面结构和导航设计
- 用例管理（查看、编辑、创建）的交互流程
- 实时执行监控的展示内容和刷新方式
- 报告在线查看和对比的功能范围
- 远程执行的架构方向（仅需求层面）
- 用户权限和安全管理（如需要）
