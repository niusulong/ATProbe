### 1.3 精简修订（用户反馈驱动）

**两次精简迭代：**

**迭代1（v1.2/v1.3）：合并数据输入方式**
- 用户反馈：原文件输入设计过度（文件类型分类、三种发送模式、行/注释/编码处理）。实际场景是"数据已以文件存储，将文件内容当作数据流发送"，只关心大文件分块。
- 决策：① 文件输入与长数据输入合并为统一数据流发送（一个 `data` 字段）；② 超阈值自动分块；③ 发送期间断连即失败（不续传）。
- 删除：文件类型分类、三种发送模式、行处理、编码处理。

**迭代2（v1.3/v1.4）：移除 HEX 字符串来源**
- 用户反馈：HEX 字符串按字符串原样发送即可，不做类型转换。
- 决策：删除 `hex` 数据来源。数据流输入只保留 `file`（文件）和 `inline`（内联/变量引用）两个来源。
- 理由：若 hex 不做十六进制→二进制转换，则与 inline 等价（都是文本发送），单独存在无意义。
- 落实：M2 §3.3 数据来源从三选一改二选一；M1 §3.2 删除 HEX 字符串来源及 HEX 解析参数；清理 §3.0 对齐说明、§3.3 对比表等交叉引用。

**当前数据流输入最终形态（两个来源）：**

```yaml
- data:
    file: firmware.bin        # 来源1：文件
- data:
    inline: "{{payload}}"     # 来源2：内联/变量引用
```

## 3. M2 v1.1 主要新增内容

### 3.1 步骤字段模型重构

步骤（step）改为**四组正交字段**（§3.1），setup/teardown/steps 共用同一结构：

| 字段组 | 字段 |
|--------|------|
| 输入方式（三选一） | `command` / `send_file` / `data` |
| 行为修饰符 | `retry` / `poll` / `when` / `timeout` / `interval` / `port` |
| 输出处理 | `extract` / `assert` |
| 失败处理 | `on_failure` |

### 3.2 新增章节

| 章节 | 内容 |
|------|------|
| §3.3 | 数据流输入（data）— file/inline 两来源 + 超阈值自动分块；衔接 M1 §3.2 |
| §4.2 | 变量断言（6 操作符）|
| §6 | 条件表达式（BNF 文法 + 求值规则，替代旧的「条件跳过」）|
| §8 | 清理步骤（teardown）|
| §9 | 重试（retry）|
| §10 | 参数化（parameters，P1 后置）|
| §11 | 轮询（poll）|
| §14.3 | name 唯一性作用域 |
| §14.4 | 正则书写规范 |
| §15 | 接口契约（与 M1）|

### 3.3 向后兼容

- 旧的单键断言 `assert: { contains: "OK" }` 仍可用（归一化为单元素列表）。
- 旧的 `when: '{{var}} == "OK"'` 仍可用（兼容期，文本替换求值）。
- 新用例推荐使用变量名引用写法。
