# M2 测试用例定义 — 审查报告

> 版本：v1.0
> 日期：2026-06-17
> 状态：评审中
> 审查对象：`docs/requirements/REQ-M2-测试用例定义.md`（v1.0，草稿）
> 关联文档：`docs/requirements/REQ-M1-串口通信管理.md`（v1.1，已确认）、`docs/requirements/PRD-总体需求.md`（v1.1）

---

## 1. 审查目的与范围

本报告从两个维度审查 M2 文档：

1. **业务场景契合度**：M2 定义的能力能否覆盖真实 AT 命令测试的典型场景。
2. **工程实现可行性**：M2 的设计是否清晰、自洽、可实现，与 M1 是否衔接顺畅。

审查基于嵌入式通信模组（蜂窝/WiFi/蓝牙）AT 命令测试的常见实践，对照成熟测试框架（Robot Framework、pytest 等）的设计经验。

---

## 2. 总体结论

M2 的骨架方向正确且现代：统一 YAML schema、三种场景共用、变量提取/引用、条件跳过、套件组织，与成熟框架理念一致。

但作为 AT 命令测试工具，**断言能力、流程控制能力偏弱，且与 M1 存在一个实质性的衔接缺口**。若直接按当前文档进入设计/实现，会在中后期暴露出大量"写不出用例"的场景。建议在定稿前完成本报告 🔴 级别问题的修订。

问题分级：

| 级别 | 含义 | 数量 |
|------|------|------|
| 🔴 严重 | 阻塞业务场景或存在设计缺陷，必须在 M2 定稿前解决 | 4 |
| 🟡 重要 | 影响易用性或覆盖面，强烈建议解决 | 5 |
| 🟢 建议 | 细节与措辞问题，修订时一并优化 | 6 |

---

## 3. 🔴 严重问题

### S1. 步骤与 M1 的「数据输入」完全没有衔接 —— 矛盾

**现状**：M1 §3 花大篇幅定义三种数据输入——直接输入（短命令）、文件输入（文本/二进制/脚本，三种发送模式）、长数据输入（内联/变量引用/HEX，分块发送）。而 M2 §3.1 步骤字段表只有 `command`，没有任何字段能表达"文件输入"或"长数据输入"。

**影响**：M1 定义的能力，在用例层面无表达方式，等于没用。固件升级、批量脚本、大报文发送这些 M1 明确点名的场景，在 M2 里写不出来。这是 M1 与 M2 的实质性矛盾。

**建议**：步骤增加数据输入字段，与 `command` 表达不同输入方式：

```yaml
steps:
  - command: AT                              # 直接输入（短命令）
  - send_file: { path: cmds.seq, mode: line }     # 文件输入-逐行
  - send_file: { path: firmware.bin, mode: chunk, chunk_size: 1024 }  # 文件输入-分块
  - data: { hex: "0102FF", append_terminator: false }  # 长数据-HEX
  - data: { inline: "{{payload}}" }          # 长数据-变量引用
```

需明确字段优先级/互斥规则（一个步骤只能有一种输入方式）。**本项涉及 M1 + M2 联动修订。**

---

### S2. 断言能力严重不足

**现状**：仅 4 种断言（contains / matches / equals / not_contains），且一个步骤只能一条断言。

**影响**（典型撞墙场景）：

1. **无法做数值范围判断**。`AT+CSQ` 返回 `+CSQ: 23,99`，"信号强度 15~31 为良好"是最常见的验证需求，当前只能写死正则 `2[3-9]|3[01]`，不可读、不可复用、无法参数化阈值。
2. **无法组合多条件**。"包含 +CEREG: 且包含 OK 且不含 ERROR"表达不了（一个 assert 只能一个键）。
3. **多行响应下 equals 几乎无用**。AT 响应几乎都是多行的且带 `\r\n`，equals 是否含尾部换行也未定义。
4. **断言失败可读性差**。matches 报错只给原始正则，用户难快速理解失败原因。

**建议**：将断言模型从"对响应原文做字符串判断"升级为**"先提取变量，再对变量做断言"**，断言主要针对 extract 出的变量，支持：

- 列表化多条件（默认 AND，支持 OR）
- 数值比较（`>` `<` `>=` `<=` `between`）
- 断言别名 `name` 字段（用于报告展示）

```yaml
steps:
  - command: AT+CSQ
    extract: { rssi: '\+CSQ:\s*(\d+)' }
    assert:
      - { name: 信号强度合格, var: rssi, op: between, min: 15, max: 31 }
      - { name: 命令成功, contains: "OK" }
      - { not_contains: "ERROR" }
```

为保持向后兼容，可保留旧的单键写法作为"对响应原文断言"的简写。

---

### S3. 变量「模板替换」进表达式，脆弱且有注入风险

**现状**：`when: '{{apn}} != ""'`、assert 里都把 `{{var}}` **文本替换**进表达式再求值。

**影响**：

1. 变量值含特殊字符（引号、空格、运算符）会破坏表达式语法甚至语义。如提取值 `a"b` → `a"b != ""` 直接出错。
2. 变量未定义/提取失败时，`{{var}}` 替换成什么（原样保留？空串？报错？）**未定义**，导致 when 行为不确定。
3. 没有类型规则：`{{count}} > 0` 隐含按数值比，`{{status}} == "READY"` 按字符串比，规则靠"有没有引号"猜测，没有明文。

**建议**：表达式里**引用变量名而非文本替换**，由求值引擎解析作用域取值：

```yaml
when: 'apn != "" and rssi > 15'      # 引用变量名，非 {{}}
```

并明确求值规则：
- 变量未定义 → 视为 null（比较一律为 false，除非用 `is null`）
- 数值比较前自动尝试类型转换，转换失败则该条件为 false
- 字符串比较需显式带引号

这避开了自己造一个易错的字符串模板替换引擎。`{{var}}` 保留用于 **command/字符串内容** 的模板替换（受控场景，风险低）。

---

### S4. 缺少步骤级「重试」，实际偶发失败无法稳定化

**现状**：`on_failure` 只有 abort/skip/continue，没有 retry。

**影响**：AT 命令测试中，单条命令**偶发超时/抖动**极为常见（网络命令尤其）。当前只能要么整条用例挂掉，要么吞掉失败，两种都不理想。setup 一律失败即跳过用例，也过于严格。

**建议**：步骤增加重试：

```yaml
- command: AT+CEREG?
  assert: { contains: "1" }
  retry: { count: 3, interval: 2000 }    # 最多重试3次，每次间隔2秒
```

setup 步骤同样支持 retry。这是成熟测试框架的标配，对 AT 测试几乎不可或缺。

## 4. 🟡 重要问题

### M1. 没有 teardown，AT 测试的「状态清理」痛点未解决

**现状**：只有 `setup`，没有 `teardown`。

**影响**：AT 测试高度依赖设备状态——建立 socket、激活 PDP、拨打电话后若不清理，会污染后续用例（端口占用、连接残留、计费等）。

**建议**：
- 增加 `teardown`，明确"用例失败也执行 teardown"。
- 建议再考虑 `suite` 级 setup/teardown（套件开头统一注网，所有用例共用，结尾统一清理）。

```yaml
teardown:
  - command: AT+CIPCLOSE
  - command: AT+CGACT=0,1
```

---

### M2. 缺少「条件循环/轮询」，无法等待异步事件

**现状**：`loop` 只能固定次数循环整个序列（压测用），无法表达条件循环。

**影响**：大量场景是"轮询直到某条件满足或超时"：等待注网（轮询 `AT+CEREG?` 直到注册成功）、等待 socket 连接建立、等待 URC。URC 虽能触发动作（M1），但 URC 触发的是"执行命令"不是"恢复用例步骤"——用例本身仍在阻塞等待。

**建议**：步骤增加轮询结构：

```yaml
- poll:
    command: AT+CEREG?
    extract: { stat: 'CEREG:\s*\d,(\d)' }
    until: 'stat == "1" or stat == "5"'
    timeout: 60        # 总超时（秒）
    interval: 3000     # 轮询间隔（毫秒）
```

到达 timeout 仍未满足 → 该步骤失败（按 on_failure 处理）。

---

### M3. 无数据驱动/参数化，相同流程不同参数只能复制用例

**现状**：M2 完全缺失参数化机制。

**影响**：测不同 APN 激活、发短信到不同号码、遍历频段组合……这些"同流程不同参数"场景，当前每个参数组合都要复制一份用例文件，维护成本高。这是成熟框架（pytest parametrize、RF DataDriver）的标配。

**建议**：支持用例参数化（参数矩阵 + 模板）。可作为 M2 后期增强，但应**在 schema 中预留字段空间**，避免后期破坏性改动。示例方向：

```yaml
parameters:
  - { apn: cmnet,  type: IP }
  - { apn: cmiot,  type: IP }

steps:
  - command: 'AT+CGDCONT=1,"{{type}}","{{apn}}"'
    assert: { contains: "OK" }
```

用例按参数组合展开为多次执行。

---

### M4. 压测的「轮次成功定义」和「失败处理」未定义

**现状**：§10.2 说命令序列压测"统计整体序列成功率"，但关键语义未定义。

**需补充明确的点**：

1. **一轮成功的标准**：序列中所有步骤都成功算一轮成功？需明文规定。
2. **压测中失败的处置**：压测 100 轮，第 50 轮某步超时——是中止整个压测，还是记一次失败继续？建议默认：记失败继续，最后统计失败率。
3. **`interval` 语义**：是"上一轮结束→下一轮开始"的间隔，还是固定节拍（不论上轮何时结束，按固定周期发）？两者结果不同，需明确。建议默认前者。
4. **预热（warmup）**：前 N 次冷启动数据会污染响应时间统计。建议支持 `warmup` 配置，预热轮次不计入统计。

---

### M5. 条件只有 skip，没有 if/else 分支

**现状**：只有"when 不满足则跳过"，要做 if-else（条件 A 发 X，否则发 Y）得写两个互斥条件的步骤，笨拙。

**建议**：至少在文档里明确"模拟 if-else"的标准写法（两个 when 互斥的步骤），或考虑支持 `else` 字段。优先级低于 S 级，但影响复杂用例的可读性。

## 5. 🟢 建议（细节与措辞）

| 编号 | 问题 | 建议 |
|------|------|------|
| G1 | §5.1 措辞"提取命名的分组"有误导，示例实际用的是普通捕获分组 `(...)` 而非命名分组 `(?P<name>...)` | 改为"提取正则的**第一个捕获分组**作为变量值"，并明确多捕获分组时取哪一个 |
| G2 | 正则转义在 YAML 里易错（`\\+CEREG:\\s*\\d` 双反斜杠） | 给出推荐写法（YAML 单引号串 + 单反斜杠），附正确示例，降低出错率 |
| G3 | `name` 声明为"唯一标识"，但套件靠文件路径引用，不同文件可能重名 | 明确 name 唯一性的**作用域**（执行范围内？套件内？），报告里用 name 还是文件路径标识用例 |
| G4 | 响应"完整性边界"未说明（断言/提取针对的"响应"从哪到哪） | 明确接口契约：M1 负责判定响应完整（收到 OK/ERROR 或超时）后交给 M2，M2 对完整响应做断言。这是与 M1 的关键接口 |
| G5 | 非编码文本（中文短信、USSD、UCS2）的 contains/extract 行为未提 | 至少说明：断言/提取针对 M1 解码后的文本，编码由 M1 处理 |
| G6 | `command` 是否含结束符未说 | 说明 command 内容不含结束符，由 M1 按 §2.1 自动追加 |

## 7. 修订优先级建议

建议按以下顺序推进 M2 修订（S1/S2 涉及 schema 主体，先定方向）：

| 顺序 | 项 | 级别 | 涉及联动 |
|------|----|------|---------|

| 4 | S3 表达式变量引用模型 | 🔴 | M2（when/assert） |
| 5 | M1 teardown / M2 轮询 | 🟡 | M1 + M2 |
| 6 | M4 压测语义补全 | 🟡 | M2 |

| 8 | G1~G6 细节措辞 | 🟢 | M2 |

## 8. 审查结论

M2 当前版本**不建议直接定稿**。需完成 S1~S4 四项严重问题修订后再进入用户确认环节。S 级问题均涉及 schema 主体设计，建议逐项讨论定方案后统一改稿，避免反复返工。

修订完成后，本审查报告状态应更新为"已闭环"，并在 `SESSION-*` 进度文档中记录本次审查与修订结果。
