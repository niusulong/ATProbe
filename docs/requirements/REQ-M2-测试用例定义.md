## 1. 模块概述

测试用例定义模块负责定义 AT 命令测试用例的格式和结构，包括用例文件格式、步骤定义、断言规则、变量提取、流程控制、用例组织等。本模块是测试执行引擎的输入，定义"测什么、怎么判断"。

本模块不依赖其他模块（定义层），但其步骤的输入方式字段与 M1 串口通信管理的数据输入能力对齐（见 §15 接口契约）。

## 2. 用例文件格式

### 2.1 统一 YAML 结构

三种测试场景（基础验证、命令序列、性能压测）共用一套 YAML schema，通过字段组合区分不同场景。

### 2.2 用例文件结构

每个用例文件定义一个测试用例，基本结构如下：

```yaml
name: 用例名称
description: 用例描述（可选）
tags: [标签1, 标签2]          # 可选，用于分类筛选
port: COM3                    # 可选，指定执行串口，不指定则使用默认串口

parameters:                   # 可选，参数化矩阵，详见 §10
  - { apn: cmnet, type: IP }

setup:                        # 可选，前置条件检查
  - command: AT
    assert: { contains: "OK" }

interval: 100                 # 可选，用例级发送间隔（毫秒），覆盖全局默认值
on_failure: continue          # 可选，用例级默认失败策略

steps:
  - command: AT+CEREG?
    extract:
      stat: 'CEREG:\s*\d,(\d)'
    when: 'stat == "1" or stat == "5"'
    assert:
      - { name: 命令成功, contains: "OK" }
    timeout: 3                # 可选，步骤级响应超时（秒）
    retry: { count: 3, interval: 2000 }

  - data:
      file: firmware.bin            # 从文件读取数据流发送，详见 §3.3
    assert: { contains: "OK" }

teardown:                     # 可选，清理步骤，详见 §8
  - command: AT+CGACT=0,1

loop:                         # 可选，存在则为压测场景，详见 §13
  count: 100
  interval: 100
```

### 2.3 场景区分规则

引擎层面按是否存在 `loop` 字段区分两种执行流程：

- **常规执行**（无 `loop` 字段）：按顺序执行 steps。涵盖基础验证和命令序列两类用例（执行流程相同，区别只在步骤间是否有变量依赖，引擎不区分）。
- **压测执行**（有 `loop` 字段）：将 steps 作为整体循环执行，统计性能指标。

**用例粒度约定：** 一个用例文件 = 一个功能点。例如测试 `AT+TEST` 指令的设置/查询/测试/边界格式，全部放在同一用例的 steps 中，而非每条命令一个用例。执行流程见 M3 §2。

### 2.4 字段说明

| 字段 | 层级 | 必填 | 说明 |
|------|------|------|------|
| name | 用例 | 是 | 用例名称，执行范围内唯一（见 §14.3） |
| description | 用例 | 否 | 用例描述 |
| tags | 用例 | 否 | 标签列表，用于分类筛选 |
| port | 用例 | 否 | 指定默认执行串口（步骤可覆盖，见 §3） |
| parameters | 用例 | 否 | 参数化矩阵，展开为多次执行（§10） |
| setup | 用例 | 否 | 前置条件检查步骤列表 |
| teardown | 用例 | 否 | 清理步骤列表，用例结束后无条件执行（§8） |
| interval | 用例/步骤 | 否 | 发送间隔（毫秒），支持用例级覆盖 |
| on_failure | 用例/步骤 | 否 | 失败策略（abort/skip/continue），支持用例级默认 + 步骤级覆盖 |
| steps | 用例 | 是 | 测试步骤列表，至少一条 |
| loop | 用例 | 否 | 压测循环配置，存在则为压测场景（§13） |

> **超时说明：** 超时只在**步骤级**配置（步骤的 `timeout` 字段）。步骤未配 timeout 时用配置文件的默认值（见 M3 §3）。不存在用例级或全局级超时继承。

## 3. 步骤定义

### 3.1 步骤字段模型

一个步骤（step）由四组**正交字段**组成，可自由组合。setup / teardown / steps 字段共用同一结构。

| 字段组 | 字段 | 说明 |
|--------|------|------|
| **输入方式**（二选一） | `command` / `data` | 发送什么，见 §3.2 ~ §3.3 |
| **行为修饰符**（可选） | `retry` / `poll` / `when` / `timeout` / `interval` / `port` | 怎么发、何时发 |
| **输出处理**（可选） | `extract` / `assert` | 提取变量、断言验证 |
| **失败处理**（可选） | `on_failure` | 步骤失败策略（§3.4） |

**约束：**

- 输入方式二选一：`command` / `data` 不可同时出现；缺省时 `command` 必填。
- `retry` 与 `poll` 互斥（语义冲突，避免双重循环）。
- teardown 步骤不支持 `when` / `retry` / `poll`（保持清理逻辑简单）。
- `port`（步骤级）覆盖用例级 `port`。同一用例的步骤可指定不同端口（双模块测试场景），步骤在指定端口串行执行，共享同一用例变量池（见 §5.3、M3 §1/§6）。

### 3.2 直接输入（command）

对应 M1 §3.1，适用于短文本命令（如 AT 指令）。

```yaml
- command: AT
  assert: { contains: "OK" }
```

- 输入内容作为字符串直接发送。
- 内容**不含结束符**，结束符由 M1 按 §2.1 配置自动追加（见 §15 接口契约）。
- 支持 `{{变量名}}` 模板替换（见 §5.2）。
- 受发送间隔（`interval`）约束。

### 3.3 数据流输入（data）

对应 M1 §3.2 数据流输入，用于发送任意字节流：可来自文件或用例内联。工具**不关心数据内部结构**（文件格式、换行、编码等），统一按字节流发送；数据量超过阈值时自动分块传输。

**设计原则（精简版）：** 文件就是数据源，发送逻辑统一走"字节流 + 超阈值自动分块"。不区分文件类型，不做逐行/注释/编码处理。

#### 3.3.1 数据来源（二选一）

通过 `file` / `inline` 两个互斥字段指定数据来源：

```yaml
# 来源1：从文件读取（数据流场景，最常用）
- data:
    file: firmware.bin          # 文件路径，支持 {{var}} 模板替换

# 来源2：用例内联 / 变量引用
- data:
    inline: "{{payload}}"       # 内联文本或变量引用
```

#### 3.3.2 字段说明

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| file | 字符串 | — | 文件路径（绝对或相对用例文件），支持 `{{var}}`；与 inline 二选一 |
| inline | 字符串 | — | 内联文本或 `{{var}}` 变量引用；与 file 二选一 |
| chunk_threshold | 数值（字节） | 4096 | 数据长度超此值时**自动启用分块**，低于则整体发送 |
| chunk_size | 数值（字节） | 1024 | 分块发送时每块的字节数 |
| chunk_interval | 数值（毫秒） | 50 | 分块发送时块间间隔 |
| append_terminator | 布尔 | false | 发送完成后是否追加命令结束符 |

#### 3.3.3 发送行为

- **整体 vs 分块**：数据长度 ≤ `chunk_threshold` 时整体发送（一次性写入）；超过阈值时按 `chunk_size` 分块，块间按 `chunk_interval` 等待。
- **字节流语义**：数据按原始字节发送，不解析内容、不转码、不处理换行。`inline` 的文本按其字符串编码（默认 UTF-8）转为字节。
- **文件读取**：`file` 字段指向的文件以二进制方式读取全部字节，作为数据流发送。文件路径不存在时，当前步骤标记失败，记录错误日志。
- **发送中断处理**：发送期间串口断连，**当前步骤直接标记失败**（不重传、不续传）。详见 §3.4 与 M1 §8。

#### 3.3.4 完整示例

```yaml
steps:
  # 固件升级：大文件自动分块传输
  - data:
      file: firmware.bin
      chunk_threshold: 4096      # 超过 4KB 自动分块（可省略，用默认值）
      chunk_size: 1024
      chunk_interval: 50
    assert: { contains: "OK" }

  # 内联数据流
  - data:
      inline: "{{upload_payload}}"
    assert: { contains: "OK" }
```

### 3.4 失败处理（on_failure）

单步失败时的处理策略：

| 策略 | 说明 |
|------|------|
| abort | 中止当前用例，标记为失败（默认） |
| skip | 跳过当前步骤，继续执行后续步骤 |
| continue | 标记当前步骤失败，继续执行后续步骤 |

失败策略可配置，默认为 abort。支持在用例级别设置默认策略（`on_failure`），也支持在步骤级别覆盖。

```yaml
on_failure: continue          # 用例级默认失败策略

steps:
  - command: AT+CSQ
    assert: { contains: "+CSQ:" }
    on_failure: skip           # 步骤级覆盖
```

**失败策略与 retry 的交互：** retry 先于 on_failure 生效。步骤先按 `retry` 重试，所有重试耗尽仍失败后，才按 `on_failure` 处理（见 §9）。

## 4. 断言规则

### 4.1 断言写法

`assert` 支持两种写法：

- **列表式（推荐）**：多个断言元素的列表，元素间默认 AND 关系。
- **单键式（兼容）**：单个断言对象，引擎归一化为"对响应原文"的单元素列表。

```yaml
# 列表式
assert:
  - { name: 命令成功, contains: "OK" }
  - { name: 不含错误, not_contains: "ERROR" }
  - { name: 信号合格, var: rssi, op: between, min: 15, max: 31 }

# 单键式（等价于列表式的单元素）
assert: { contains: "OK" }
```

### 4.2 断言元素类型

**A. 响应原文断言**（针对完整响应文本，沿用旧语义）：

| 操作 | 键 | 语义 | 示例 |
|------|----|------|------|
| 包含 | `contains` | 响应包含指定字符串 | `{ contains: "OK" }` |
| 不包含 | `not_contains` | 响应不包含指定内容 | `{ not_contains: "ERROR" }` |
| 正则匹配 | `matches` | 响应匹配正则表达式 | `{ matches: "\\+CEREG:.*1" }` |
| 完全相等 | `equals` | 响应完全等于指定内容（含尾部换行） | `{ equals: "OK\r\n" }` |

**B. 变量断言**（针对 extract 出的变量，新增）：

| 操作符 `op` | 语义 | 示例 |
|------------|------|------|
| `eq` / `ne` | 等于 / 不等于 | `{ var: status, op: eq, value: "READY" }` |
| `gt` / `lt` / `ge` / `le` | 数值比较 | `{ var: rssi, op: ge, value: 15 }` |
| `between` | 数值闭区间 | `{ var: rssi, op: between, min: 15, max: 31 }` |
| `in` | 值在集合中 | `{ var: stat, op: in, values: ["1", "5"] }` |
| `contains` | 字符串包含 | `{ var: urc, op: contains, value: "cmnet" }` |
| `matches` | 正则匹配 | `{ var: imei, op: matches, value: "^\\d{15}$" }` |

变量断言通过 `var` 指定变量名，`op` 指定操作符，其余键为操作参数（`value`/`min`/`max`/`values`）。

### 4.3 断言行为

- 每个步骤可指定多条断言（列表式），元素间默认 **AND** 关系，全通过 = 步骤断言通过。
- 任一元素失败 → 步骤断言失败。
- 未指定 `assert` 的步骤仅发送命令，不验证响应（用于纯操作步骤如延时、设置等）。
- 断言针对的"响应"边界由 §15 接口契约定义（M1 判定响应完整后交给 M2）。

### 4.4 断言失败可读性

- 每个断言元素可选 `name` 字段（如"信号强度合格"），用于报告展示。
- 缺省时由引擎生成（如 `contains:OK`）。
- 报告展示每个断言元素：name + 期望 + 实际值（变量断言展示变量实际值，类型转换失败展示原因）。

### 4.5 断言求值与异常处理

- 变量未定义或类型转换失败 → 该断言元素**判为失败**（不抛异常），记录"变量未定义/类型不符"。
- 数值类操作（`gt`/`lt`/`ge`/`le`/`between`）对变量值尝试转数值，失败则该元素失败。
- 类型转换规则与条件表达式一致（见 §6）。

## 5. 变量系统

### 5.1 变量提取

通过正则表达式从命令响应中提取变量：

```yaml
steps:
  - command: AT+CEREG?
    assert: { contains: "+CEREG:" }
    extract:
      stat: 'CEREG:\s*\d,(\d+)'        # 提取第一个捕获分组
      lac: 'CEREG:\s*\d+,"([0-9A-F]+)"' # 每条 extract 提取各自第一个捕获分组
```

- 提取正则的**第一个捕获分组**作为变量值。
- 一个 `extract` 键对应一条正则；若需提取多个值，写多条 extract（不同正则或不同变量名）。
- 提取失败（正则无匹配）时，变量值为空，变量标记为"已提取但空值"（区别于"未定义"）。
- 命名分组 `(?P<name>...)` 不单独处理，统一取第一个捕获分组。

### 5.2 变量引用

变量在两种场景被使用，规则不同：

| 场景 | 语法 | 求值方式 | 适用字段 |
|------|------|---------|---------|
| **字符串内容模板** | `{{变量名}}` | 文本替换，值原样嵌入 | `command`、`data.inline`、`data.file` |
| **条件表达式** | 裸变量名 | 引擎按作用域取值，类型安全比较 | `when`、`poll.until` |

**字符串模板示例：**

```yaml
steps:
  - command: 'AT+CIPSTART="TCP","{{server_ip}}",8080'   # {{}} 文本替换
    assert: { contains: "CONNECT" }
```

**条件表达式示例：**

```yaml
steps:
  - command: 'AT+CGACT=1,1'
    when: 'apn != "" and stat == "1"'   # 裸变量名，引擎求值
```

变量未定义/提取失败（空值）的求值行为见 §6 表达式求值规则。

### 5.3 变量作用域

**唯一作用域：用例级变量池。** 变量属于当前用例，生命周期 = 用例生命周期。用例结束时变量池销毁。

- 变量在当前用例的所有步骤间共享（包括 setup/steps/teardown，包括跨端口步骤）。
- **变量属于用例，不属于端口**。无论步骤在哪个端口执行，都读写同一个用例变量池。
- 用例之间不共享变量，每个用例有独立的变量池。
- 步骤串行执行（见 M3 §1/§6），前一步骤的 extract 一定在后一步骤开始前完成，变量可见性天然保证，无竞态问题。

**写入与覆盖：**
- 参数化参数（§10，若启用）在用例开始时注入变量池（初始值）。
- 步骤的 extract 立即写入变量池，后续步骤立即可见。
- 同名变量后赋值覆盖（含覆盖参数化参数）。

**环境配置兜底层（M7 引入）：** 除用例级变量池外，还存在一层**全局只读的环境配置**（M7），承载跨用例共享的固化参数（FTP/HTTP 服务器地址、FOTA 版本号等）。

- 引用语法：环境配置用**点号引用** `{{group.param}}`（如 `{{ftp.host}}`、`{{fota.version_a}}`）。
- 查找优先级：用例级变量池（extract/parameters，简单名）> 环境配置（点号名）> 未定义报错。
- **点号名 `{{group.param}}` 只查环境配置**，不被 extract 覆盖（保证环境配置全局只读语义）；用例级动态值用简单名变量另存。
- 环境配置在引擎启动时一次性加载（M3），执行期间只读。详见 M7。

> **不存在 export 机制，也不存在跨用例的变量传递。** 跨端口的联动场景（如双模块测试：服务端+客户端）通过**同一用例内步骤分布在不同端口**实现，共享该用例的变量池。见下例。

**跨端口步骤共享变量示例（双模块测试）：**

一个用例的步骤分布在不同端口，共享同一个用例变量池：

```yaml
name: 双模块-TCP通信测试
steps:
  # 步骤1：COM3（服务端）建立服务器并查询IP
  - command: AT+CIPSERVER=1,8080
    port: COM3
    assert: { contains: "OK" }
  - command: AT+CIFSR
    port: COM3
    extract: { server_ip: '(\d+\.\d+\.\d+\.\d+)' }   # 提取IP写入用例变量池

  # 步骤2：COM5（客户端）连接服务端，引用变量池中的IP
  - command: 'AT+CIPSTART="TCP","{{server_ip}}",8080'
    port: COM5
    assert: { contains: "CONNECT" }
```

执行时步骤在 COM3→COM5 间串行切换，全程共享一个用例变量池。执行流程见 M3 §2/§6。

### 5.4 内置变量

| 变量名 | 说明 |
|--------|------|
| `{{timestamp}}` | 当前时间戳 |
| `{{port}}` | 当前执行的串口标识 |
| `{{loop_index}}` | 当前循环索引（压测场景，从 1 开始） |

内置变量用于字符串模板。在条件表达式中引用需先 extract 到普通变量。

## 6. 条件表达式

### 6.1 适用范围

条件表达式统一用于 `when`（步骤条件跳过）和 `poll.until`（轮询终止条件），共用同一套语法和求值规则。

### 6.2 表达式语法

```
表达式  := 或表达式
或表达式 := 与表达式 ( 'or' 与表达式 )*
与表达式 := 比较表达式 ( 'and' 比较表达式 )*
比较表达式 := 操作数 运算符 操作数  |  操作数 'is' 'null'  |  操作数 'is' 'not' 'null'
操作数   := 变量名 | 字符串字面量 | 数值字面量
运算符   := == | != | > | < | >= | <=
```

- 字符串字面量：双引号包裹，如 `"READY"`
- 数值字面量：纯数字，如 `15`
- 变量名：裸写（不加 `{{}}`），引擎按作用域取值

### 6.3 求值规则

1. **变量取值**：从用例变量池解析（含参数化参数、extract 值）。未定义 → null。
2. **null 比较**：除 `is null` / `is not null` 外，含 null 的比较一律为 **false**。
3. **类型规则**：
   - `==` / `!=`：按字符串比较（值转字符串后比较）
   - `>` / `<` / `>=` / `<=`：按数值比较，两侧自动尝试转数值，失败则该比较为 false
4. **变量值与字面量比较**：变量侧自动向字面量类型靠拢（字面量是数值则尝试转数值，是字符串则按字符串）。
5. **提取失败（空值）**：变量值为空字符串，参与比较时按空字符串处理（非 null）。

### 6.4 示例

```yaml
steps:
  - command: AT+CEREG?
    extract: { stat: 'CEREG:\s*\d,(\d)' }
    when: 'stat == "1" or stat == "5"'      # 字符串比较
  - command: AT+CSQ
    extract: { rssi: '\+CSQ:\s*(\d+)' }
    when: 'rssi > 15'                        # 数值比较
  - command: AT+CIPSEND=...
    when: 'peer_ip is not null'              # null 判断
  - command: AT+COPS?
    when: 'stat == "1" and rssi >= 15'       # AND 组合
```

### 6.5 兼容旧写法

旧写法 `when: '{{var}} == "OK"'` 仍可用——引擎检测到 `{{}}` 时，先做文本替换再求值（兼容期）。**不推荐用于新用例**，新用例请用变量名引用。

### 6.6 模拟 if-else

`when` 只能跳过步骤，不支持 else。模拟 if-else 用两个 when 互斥的步骤：

```yaml
steps:
  - command: AT+CIPSEND=10
    when: 'mode == "online"'
    assert: { contains: "OK" }
  - command: AT+CIPSEND=0
    when: 'mode != "online"'      # 与上一步互斥
    assert: { contains: "OK" }
```

## 7. 前置条件检查（setup）

### 7.1 定义方式

前置条件是命令检查式：发送命令并验证响应，失败则跳过整个用例。

```yaml
setup:
  - command: AT
    assert: { contains: "OK" }
    retry: { count: 3, interval: 1000 }    # setup 步骤支持 retry
  - command: AT+CPIN?
    assert: { contains: "READY" }
```

### 7.2 行为规则

- 前置条件在用例步骤之前执行。
- 任一前置条件失败（含 retry 耗尽），跳过整个用例（标记为"跳过"而非"失败"）。
- 前置条件中也可提取变量，供后续步骤使用。
- 前置条件为简化检查，不支持 `when`、`on_failure` 等字段，**支持 `retry`**（修正"setup 一律失败即跳过过于严格"的问题）。
- 前置条件失败，teardown 仍会执行（见 §8）。

## 8. 清理步骤（teardown）

### 8.1 定义方式

teardown 定义用例结束后的清理步骤（如关闭连接、释放 PDP、复位状态）。

```yaml
steps:
  - command: AT+CIPSTART="TCP","{{ip}}",8080
    assert: { contains: "CONNECT" }
teardown:                          # 无论用例成功/失败/跳过都执行
  - command: AT+CIPCLOSE
  - command: AT+CGACT=0,1
```

### 8.2 行为规则

- teardown 在用例**结束后无条件执行**：用例成功、失败、被 setup 跳过均执行（进程异常崩溃除外）。
- teardown 步骤的失败**不影响用例结果**（用例结果已定），仅记录警告日志。
- teardown 步骤**不支持** `when` / `retry` / `poll` / `on_failure`，保持清理逻辑简单。
- teardown 仍可提取变量（一般无用，但保留能力）。

## 9. 重试（retry）

### 9.1 定义方式

步骤增加 `retry` 修饰符，setup / steps 通用（teardown 不支持 retry）。

```yaml
steps:
  - command: AT+CEREG?
    assert: { var: stat, op: in, values: ["1", "5"] }
    extract: { stat: 'CEREG:\s*\d,(\d)' }
    retry:
      count: 3            # 最大重试次数（不含首次），默认 0（不重试）
      interval: 2000      # 重试间隔(ms)，默认 0
```

### 9.2 执行语义

1. 首次执行失败（断言失败、响应超时）→ 按 `retry.interval` 等待 → 重试。
2. 重试上限：首次 + `count` 次重试。例如 `count: 3` 共执行最多 4 次。
3. 任一次成功 → 步骤成功，停止重试。
4. 全部失败 → 步骤失败，按 `on_failure` 处理。
5. **重试粒度**：每次重试是"重新发送命令 + 重新断言 + 重新 extract"的完整步骤执行（extract 在每次执行中重算，保留最后一次成功的变量值）。
6. 重试期间若触发串口重连（M1 §4.2），重连成功后计入重试次数。
7. `retry` 与 `poll` 互斥。

## 10. 参数化（parameters）

> **落地策略**：P1 增强功能，schema 已定义，实现可后置。不阻塞 M2 主流程定稿。

### 10.1 定义方式

用例增加 `parameters` 字段（参数矩阵），用例按行展开为多次执行。

```yaml
name: PDP激活-多APN
parameters:
  - { apn: cmnet,  type: IP }
  - { apn: cmiot,  type: IP }
  - { apn: '',      type: IPV6 }
steps:
  - command: 'AT+CGDCONT=1,"{{type}}","{{apn}}"'
    assert: { contains: "OK" }
```

### 10.2 行为规则

- `parameters` 是列表，每个元素是一个参数字典。
- 用例执行时按列表展开为 N 次独立执行，每次参数注入到**用例级变量作用域**（最高优先级，见 §5.3）。
- `{{param}}` 在 command/data/path 中做字符串模板替换。
- 报告中每次执行作为独立用例实例展示（name 加后缀 `#1`/`#2`）。
- 参数化用例的 extract 变量与参数变量同作用域，extract 提取的同名变量覆盖参数值（后赋值覆盖）。

## 11. 轮询（poll）

### 11.1 定义方式

步骤增加 `poll` 修饰符，用于等待异步事件（如等待注网、等待连接建立）。

```yaml
steps:
  - command: AT+CEREG?
    extract: { stat: 'CEREG:\s*\d,(\d)' }
    poll:
      until: 'stat == "1" or stat == "5"'   # 条件表达式（用 §6 模型）
      timeout: 60                            # 总超时(秒)，必填
      interval: 3000                         # 轮询间隔(ms)，默认 1000
    on_failure: continue                      # 超时未满足→按此策略
```

### 11.2 行为规则

1. 执行命令 → extract 变量 → 求 `until` 表达式。
2. 满足 → 步骤成功，停止。
3. 不满足 → 等 `interval` → 再次执行命令（重新 extract）。
4. 累计达 `timeout` 仍未满足 → 步骤失败，按 `on_failure`。
5. poll 期间每次执行的响应都记录到原始日志；报告中展示轮询次数和总耗时。
6. poll 与 retry 互斥（语义重叠，避免双重循环）。

## 12. 用例套件

### 12.1 套件定义

套件是独立的 YAML 文件，通过路径引用用例文件：

```yaml
name: 网络注册测试套件
description: 网络注册相关用例集合
tags: [network, regression]

suite_setup:                       # 可选，套件开头执行一次（P1 增强功能）
  - command: AT+CFUN=1
suite_teardown:                    # 可选，套件结尾执行一次（P1 增强功能）
  - command: AT+CFUN=0

cases:
  - network-basic_register.yaml
  - network-pdp_activate.yaml
  - network-data_transfer.yaml
```

### 12.2 套件功能

- 按顺序执行引用的用例。
- 支持按标签筛选执行套件中的部分用例。
- 套件内用例之间独立，不共享变量。

**套件级前后置执行顺序（P1 增强功能）：**

```
suite_setup → [用例setup → 用例steps → 用例teardown]×N → suite_teardown
```

### 12.3 命名规范

**用例文件命名：**

```
{功能域}-{测试点}.yaml
```

- 全小写英文字母，单词间用下划线分隔
- 以功能域开头，体现归属
- 示例：`network-basic_register.yaml`、`sms-send_text.yaml`、`stress-loop_at.yaml`

**套件文件命名：**

```
suite-{功能域}.yaml
```

- 以 `suite-` 前缀区分套件和用例文件
- 示例：`suite-network.yaml`、`suite-sms.yaml`、`suite-smoke.yaml`

**用例 name 字段：**

- 用例文件内 `name` 字段支持中文，便于在报告和日志中展示
- 建议格式：`{功能点}-{测试场景}`，如 `网络注册-基础验证`、`短信-发送文本短信`

**套件 name 字段：**

- 同样支持中文
- 建议格式：`{功能域}测试套件`，如 `网络注册测试套件`、`冒烟测试套件`

### 12.4 目录结构

按功能块组织，每个功能块一个目录，套件和用例放在一起：

```
testcases/
├── network/                            # 网络功能块
│   ├── suite-network.yaml              # 该功能的套件
│   ├── network-basic_register.yaml
│   ├── network-pdp_activate.yaml
│   └── network-data_transfer.yaml
├── sms/                                # 短信功能块
│   ├── suite-sms.yaml
│   ├── sms-send_text.yaml
│   └── sms-receive_text.yaml
└── stress/                             # 压测功能块
    ├── suite-stress.yaml
    └── stress-loop_at.yaml
```

**目录规则：**

- 每个功能域一个顶层目录，目录名即功能域
- 套件与用例文件放在同一目录，就近维护
- 套件中引用用例使用同目录相对路径，如 `network-basic_register.yaml`
- 功能域目录名称与用例文件前缀保持一致

## 13. 压测配置

### 13.1 基本配置

```yaml
loop:
  count: 100                   # 循环次数（必填）
  interval: 100                # 循环间隔（毫秒，可选，默认0）
  warmup: 5                    # 预热轮数，默认 0
  abort_on_failure: false      # 遇失败是否中止，默认 false
```

### 13.2 压测语义（明文定义）

| 项目 | 定义 |
|------|------|
| **一轮成功标准（单命令压测）** | 该命令断言通过 = 成功 |
| **一轮成功标准（命令序列压测）** | 一轮内所有步骤断言全通过 = 该轮成功 |
| **压测中失败的处置** | 默认**记一次失败继续**（不中止整个压测），最终统计失败率。设 `abort_on_failure: true` 则遇失败即止 |
| **interval 语义** | **"上一轮结束 → 下一轮开始"的间隔**（非固定节拍） |
| **warmup 语义** | 前 N 轮**执行但不计入响应时间统计**，避免冷启动污染数据 |

### 13.3 统计维度

- 单命令压测：统计该命令的响应时间分布和成功率。
- 命令序列压测：统计每步的响应时间分布和成功率 + 整体序列成功率。
- 响应时间分布指标：min / max / avg / P95 / P99。

## 14. 标签系统与正则规范

### 14.1 标签定义

- 纯自定义标签，用户自由定义任意字符串
- 标签在用例和套件中通过 `tags` 字段声明

```yaml
tags: [smoke, network, p0]
```

### 14.2 标签使用

- 执行时通过标签筛选：只执行匹配标签的用例
- 报告中按标签分组统计
- 支持多标签组合筛选（交集/并集）
- 一个用例可有多个标签

### 14.3 name 唯一性作用域

- `name` 在**单个执行范围内**（一次运行加载的所有用例）必须唯一。
- 报告中以"文件名 + name"双标识展示用例，避免不同文件重名混淆。
- 套件靠文件路径引用用例，不依赖 name 唯一性做引用。

### 14.4 正则书写规范

YAML 中正则表达式易因反斜杠转义出错（`\\+CEREG:\\s*\\d` 双反斜杠难以阅读）。

**推荐写法：用单引号字符串**，正则内反斜杠**单写一次**：

```yaml
# ✅ 推荐：单引号 + 单反斜杠
extract:
  stat: '\+CEREG:\s*\d,(\d)'

# ❌ 不推荐：双引号导致反斜杠需双写
extract:
  stat: "\\+CEREG:\\s*\\d,(\\d)"
```

**原因**：YAML 单引号字符串中反斜杠无特殊含义，原样保留；双引号字符串中反斜杠是转义符，需双写。统一用单引号可显著降低正则出错率。

## 15. 接口契约（与 M1）

### 15.1 响应完整性判定责任划分

- **M1 负责**判定响应完整：收到 OK / ERROR / URC 终结标志，或响应超时。M1 将完整响应文本交给 M2。
- **M2 负责**对完整响应做 extract 和 assert。
- M2 层面不感知"响应是否完整"，只面对 M1 交付的完整响应文本。

### 15.2 编码处理责任划分

- extract / assert 针对 M1 **解码后的文本**操作。
- 中文短信、USSD、UCS2 等编码转换由 M1 处理，M2 层面只面对解码后的文本。

### 15.3 结束符处理

- `command` 字段内容**不含结束符**，结束符由 M1 按 §2.1 配置自动追加。
- `data` 字段是否追加结束符由 `append_terminator` 控制（见 §3.3）。

### 15.4 数据输入参数对齐

M2 步骤的输入方式字段（`command` / `data`）参数与 M1 §3 数据输入参数完全对齐，详见各字段说明（§3.2 ~ §3.3）。M1 侧参数命名见 M1 §3 对应章节。
