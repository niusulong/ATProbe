## 1. 模块概述

### 1.1 解决的问题

网络相关功能（FTP / HTTP / TCP / MQTT）和 FOTA（固件升级）测试中，存在一类**跨用例共享、相对静态、人工预设**的配置参数：

- **传输服务器信息**：FTP 的 host/port/user/pass、HTTP 的 host/port、MQTT 的 broker/port 等。
- **FOTA 包信息**：升级包名称、升级前版本号、升级后版本号（往返升级时是两组）。
- **设备/账户信息**：设备型号、测试号码、APN 等。

这些参数的特征：

| 特征 | 说明 |
|------|------|
| **跨用例共享** | 同一个 FTP 服务器地址被多个 FTP 用例引用；同一个 FOTA 包信息被 FOTA-over-FTP 和 FOTA-over-HTTP 用例共用 |
| **相对固化** | 测试环境稳定后即固化，换环境（如换测试服务器）才改 |
| **每次测试可能不同** | 不同测试阶段、不同被测设备，IP/端口/版本号都可能变化 |
| **可扩展** | 用户可自由增加新的配置组和参数 |

**用例的 `{{var}}` 模板替换**在查找时，若用例级变量池（extract/parameters）未命中，**回退到环境配置**读取。环境配置是测试的"全局默认值字典"。

### 1.2 与现有变量体系的关系（关键，避免概念混淆）

本模块**不改变**M2 §5.3「变量作用域仅用例级」的既有决策。环境配置是一层**正交的、只读的兜底层**：

| | 环境配置（本模块） | 用例级变量（M2 §5.3） |
|---|---|---|
| 产生方式 | 人工预设（配置文件/UI 编辑） | 执行中 extract 提取 / parameters 注入 |
| 作用域 | **全局**，所有用例共享 | 仅当前用例 |
| 可变性 | **只读**（执行中不写，仅读取填充） | 可写（extract 覆盖） |
| 生命周期 | 跨用例持久，不随用例结束销毁 | 用例结束即弃 |
| 引用语法 | `{{group.param}}`（点号引用） | `{{var}}`（简单名） |
| 典型用途 | 服务器地址、版本号、测试号码等固化参数 | 上一步响应里提取的动态值（如实际分配的 IP） |

**冲突解决（同名优先级）：** 模板替换查找顺序——

```
1. 用例级变量池（extract / parameters）   ← 命中即用，最高优先级
2. 环境配置（env.yaml / UI 配置）          ← 兜底默认值
3. 都未命中 → 未定义错误，步骤 FAIL（报错：变量 xxx 未定义）
```

> 这样设计的好处：环境配置给"默认值"，用例可用 extract 临时覆盖（如某用例先查询实际 IP 再用，覆盖环境配置里的静态 IP）。

### 1.3 跨域共享的核心难点（需求方明确场景）

需求方明确指出：**FOTA 包信息无论走 FTP 还是 HTTP 升级都是同一份；但 FTP 服务器信息和 HTTP 服务器信息不同。**

这决定了配置组织不能简单按"传输类型"分组，而要按**职责域**分组，让不同传输用例自由组合引用：

```
FOTA-over-FTP 用例  → 引用 {{ftp.host}}（传输） + {{fota.pkg_ab}}（包）
FOTA-over-HTTP 用例 → 引用 {{http.host}}（传输） + {{fota.pkg_ab}}（同一份包配置）
```

因此本模块采用**按职责域分组 + 点号跨组引用**的模型（见 §3）。

### 1.4 核心设计原则

- **单一全局配置**：一份配置文件定义当前测试环境的全部参数（不支持多环境切换，换环境改文件，见 §2）。
- **分组结构化**：参数按职责域分组（ftp/http/tcp/mqtt/fota/device 等），组内键值对。
- **点号引用**：用例中 `{{group.param}}` 引用，跨组自由组合。
- **只读 + 用例级优先**：执行中只读取填充，不改写；同名时用例级 extract 优先。
- **可扩展**：用户可任意新增组、新增参数，无需改代码或 schema。
- **人工编辑 + UI 编辑双入口**：配置文件 YAML 可直接编辑，UI 提供可视化编辑面板。

## 2. 配置文件

### 2.1 文件位置与命名

- 默认文件名：`env.yaml`，位于工作目录（与 `atprobe.yaml` 同级）。
- CLI 可用 `--env-config <路径>` 覆盖（M5 新增选项，见 §6）。
- UI 在设置面板指定当前环境配置文件路径（M6 新增面板，见 §7）。

### 2.2 文件格式（YAML，单一全局配置）

```yaml
# env.yaml — 测试环境配置（单一全局，换环境改此文件）
# 结构：顶层键 = 职责域（组），组内键值对 = 参数

ftp:
  host: 192.168.1.100
  port: 21
  user: test
  password: test123
  path: /firmware

http:
  host: 192.168.1.200
  port: 8080
  base_url: /api

tcp:
  host: 192.168.1.100
  port: 8080

mqtt:
  broker: 192.168.1.150
  port: 1883
  client_id: atprobe_test
  topic: test/cmd

fota:
  # 往返升级：存绝对标识，用例按方向绑定 before/after
  version_a: V1.0.0          # 版本 A
  version_b: V2.0.0          # 版本 B
  pkg_ab: fota_V1_to_V2.bin  # A→B 升级包
  pkg_ba: fota_V2_to_V1.bin  # B→A 回退包
  # FOTA-over-FTP 和 FOTA-over-HTTP 共用这份包配置，
  # 各自用例再引用 {{ftp.*}} 或 {{http.*}} 的传输信息

device:
  model: EC600S
  test_number: "13800138000"   # 字符串，避免被解析为数值丢失精度
  apn: cmnet
```

### 2.3 格式规则

- 顶层键即**组名**（职责域），组名由用户自定义，无固定枚举。
- 组内为键值对，值支持：字符串、数值、布尔。建议服务器端口用数值、号码类用字符串（加引号）。
- **不支持嵌套组**（组内不再嵌组），保持扁平键值对，引用路径固定两级 `{{group.param}}`。
- 文件可有注释（`#`），便于标注每个参数用途。
- 组和参数都可任意增删，无 schema 强约束（可选校验，见 §8）。

### 2.4 加载时机

- 引擎 `start(配置)` 时（M3），由 CLI/UI 把 env 配置路径传入；M3 加载解析后注入**用例模板替换上下文**。
- 加载失败（文件不存在/语法错误）→ 引擎 ERROR（M3 §7.5 场景C），提示行号原因，不执行任何用例。
- 加载在所有用例执行前一次性完成，执行期间不重载（只读语义）。

## 3. 参数组织模型

### 3.1 按职责域分组

组是**职责域**，不是传输类型。关键区分：

| 组 | 职责 | 共享情况 |
|----|------|---------|
| `ftp` | FTP 传输服务器信息 | 仅 FTP/FOTA-over-FTP 用例引用 |
| `http` | HTTP 传输服务器信息 | 仅 HTTP/FOTA-over-HTTP 用例引用 |
| `tcp` | TCP 服务器信息 | TCP 用例引用 |
| `mqtt` | MQTT broker 信息 | MQTT 用例引用 |
| `fota` | FOTA 包信息（与传输无关） | **被所有 FOTA 用例共享**，无论走哪种传输 |
| `device` | 设备/账户通用信息 | 任意用例引用 |
| *自定义* | 用户按需新增 | — |

> FOTA 组独立存在，是因为"包信息"与"用什么传输下载"是正交的两个维度。

### 3.2 引用语法（点号引用）

用例中通过 `{{group.param}}` 引用环境配置：

```yaml
# FOTA-over-FTP 升级用例（A→B）
name: FOTA-FTP升级-A到B
steps:
  - command: 'AT+QFTPURL="{{ftp.host}}",{{ftp.port}}'      # 引用 ftp 组
    assert: { contains: "OK" }
  - command: 'AT+QFOTAPKG="{{fota.pkg_ab}}"'               # 引用 fota 组（跨组）
    assert: { contains: "OK" }
  - command: 'AT+QFWVER?'
    extract: { cur_ver: 'FWVER:\s*(\S+)' }
    assert:
      - { name: 升级前版本, equals: "{{fota.version_a}}" }  # 升级前校验
  # ... 触发升级 ...
  - command: 'AT+QFWVER?'
    extract: { new_ver: 'FWVER:\s*(\S+)' }
    assert:
      - { name: 升级后版本, equals: "{{fota.version_b}}" }  # 升级后校验
```

```yaml
# FOTA-over-HTTP 升级用例（A→B）—— 同一份 fota 包配置，换传输组
name: FOTA-HTTP升级-A到B
steps:
  - command: 'AT+QHTTPURL="http://{{http.host}}:{{http.port}}/{{fota.pkg_ab}}"'  # http + fota 跨组
    assert: { contains: "OK" }
  - command: 'AT+QFWVER?'
    assert:
      - { equals: "{{fota.version_b}}" }
```

### 3.3 往返升级的配置策略

FOTA 常需往返（A→B→A）验证。配置存**绝对标识**（不存"前/后"这种相对方向），每个用例按自己的方向绑定：

| 配置项 | 值 | 含义 |
|--------|-----|------|
| `fota.version_a` | V1.0.0 | 版本 A（绝对） |
| `fota.version_b` | V2.0.0 | 版本 B（绝对） |
| `fota.pkg_ab` | fota_V1_to_V2.bin | A→B 包 |
| `fota.pkg_ba` | fota_V2_to_V1.bin | B→A 包 |

- 升级用例（A→B）：before = `{{fota.version_a}}`，after = `{{fota.version_b}}`，pkg = `{{fota.pkg_ab}}`。
- 回退用例（B→A）：before = `{{fota.version_b}}`，after = `{{fota.version_a}}`，pkg = `{{fota.pkg_ba}}`。

> 用"绝对标识 + 用例绑定方向"而非"前/后版本"，是因为同一个版本在升级用例里是"后"、在回退用例里是"前"，存绝对值避免歧义。

### 3.4 引用规则

- 引用必须形如 `{{group.param}}`，两级路径。
- 引用未定义的组或参数 → 步骤 FAIL，错误信息明确（如 `环境配置未定义: fota.version_c`）。
- 环境配置引用与用例级变量引用（`{{var}}`）共存于同一模板，按 §1.2 优先级查找。
- 支持在命令、断言 expected、data 模板、when 条件中引用（与 M2 §5 变量引用位置一致）。

## 4. 与用例变量的协同

### 4.1 查找优先级（落地 M2 §5.3 的补充）

模板替换时，对每个 `{{xxx}}` 按以下顺序查找：

```
1. 若 xxx 含点号（形如 group.param）→ 仅查环境配置（用例级变量不使用点号名）
   └─ 命中 → 用环境配置值
   └─ 未命中 → 未定义错误
2. 若 xxx 不含点号（简单名）→ 先查用例级变量池
   ├─ 命中 → 用用例级值（extract/parameters 优先）
   └─ 未命中 → 查环境配置的"默认组"（可选，见 §4.2）
       ├─ 命中 → 用环境配置值
       └─ 未命中 → 未定义错误
```

### 4.2 默认组（可选简化）

为方便，可约定一个**默认组**（如 `default` 或根级参数），简单名 `{{param}}` 在用例级未命中时回退到默认组查找。是否启用默认组在 §8 待细化，第一阶段可不实现（强制全用 `{{group.param}}`）。

### 4.3 用例级覆盖环境配置

用例可在 setup 或早期步骤用 extract 提取实际值，覆盖（遮蔽）环境配置：

```yaml
# 环境配置 ftp.host=192.168.1.100（静态默认）
# 但本用例要先查询设备实际获取的 FTP 服务器地址
setup:
  - command: AT+QFTPGET?
    extract: { "ftp.host": '"host","([^"]+)"' }   # 提取后写入用例级，遮蔽环境配置
steps:
  - command: 'AT+QFTPURL="{{ftp.host}}"'   # 这里用点号引用——注意见 §4.4
```

### 4.4 点号名的 extract 约定（重要边界）

由于点号引用 `{{group.param}}` 默认指向环境配置，若用例想用 extract 覆盖某个点号名，需明确：

- **第一阶段规则**：extract 写入用例级变量池用**简单名**（不带点号），点号引用 `{{group.param}}` **只查环境配置**，不可被 extract 覆盖。
- 即：环境配置是纯只读默认值，用例级动态值用简单名变量另存，两者不混用点号命名空间。

> 这条边界保证环境配置的"全局只读默认值"语义清晰，避免用例执行意外改写全局配置造成后续用例污染。

## 5. 可扩展性

### 5.1 用户自定义组与参数

- 用户新增功能域（如新增 `coap`、`lwm2m` 测试）→ 在 `env.yaml` 加一个组即可，用例用 `{{coap.host}}` 引用。
- 无需改任何代码、schema、CLI/UI 逻辑——加载器遍历所有顶层组，UI 按组动态渲染编辑面板（§7）。
- 组名、参数名由用户自定义，遵循 YAML 标识符规范（字母数字下划线）。

### 5.2 预置组（建议，非强制）

第一阶段建议预置这些组作为示例（用户可改可删）：

| 组 | 典型参数 | 用途 |
|----|---------|------|
| `ftp` | host/port/user/password/path | FTP 测试 |
| `http` | host/port/base_url | HTTP 测试 |
| `tcp` | host/port | TCP 测试 |
| `mqtt` | broker/port/client_id/topic | MQTT 测试 |
| `fota` | version_a/version_b/pkg_ab/pkg_ba | FOTA 往返升级 |
| `device` | model/test_number/apn | 设备通用 |

> 这些只是示例起点，不构成强制 schema。

## 6. CLI 集成（M5 补充）

M5 新增选项：

| 选项 | 说明 | 默认 |
|------|------|------|
| `--env-config <路径>` | 指定环境配置文件 | `./env.yaml` |

- M5 `atprobe.yaml` 配置文件可设 `env_config: ./env.yaml` 作为默认路径（命令行 `--env-config` 覆盖）。
- `run` 子命令启动时把环境配置路径传给 M3 引擎（M3 加载）。
- `list` 子命令可加 `env` 子目标，列出当前环境配置的组和参数（便于核对）。
- 不提供 `--env-config` 且默认 `env.yaml` 不存在 → **不视为错误**，环境配置层为空（所有点号引用都会未定义报错，除非用例不用点号引用）。这保证不依赖环境配置的用例正常跑。

## 7. UI 集成（M6 补充）

M6 新增「环境配置」面板（侧边栏新增导航项，或作为独立选项卡类型 `env_config`）：

### 7.1 面板功能

```
┌─ 环境配置  文件: [./env.yaml ▼] [加载] [另存为] ─────────┐
│ ☑ 启用环境配置（取消则执行时不加载）                      │
├──────────────────────────────────────────────────────────┤
│ ▼ ftp                    [＋参数] [删除组]                │
│   host       [192.168.1.100        ]                     │
│   port       [21                   ]                     │
│   user       [test                 ]                     │
│   password   [••••••               ] 👁                  │
│ ▼ fota                    [＋参数] [删除组]                │
│   version_a  [V1.0.0               ]                     │
│   version_b  [V2.0.0               ]                     │
│   pkg_ab     [fota_V1_to_V2.bin    ]                     │
│ ▼ device                  [＋参数] [删除组]                │
│   ...                                                     │
│ [＋新建组]                                               │
├──────────────────────────────────────────────────────────┤
│ [保存]  [重新加载文件]  [校验引用]                        │
└──────────────────────────────────────────────────────────┘
```

### 7.2 编辑能力

| 能力 | 说明 |
|------|------|
| 分组展示 | 按组折叠展示，组内键值对表格编辑 |
| 增删组 | `＋新建组` / 组的 `删除组`，动态增减 |
| 增删参数 | 组内 `＋参数` / 行级删除 |
| 密码脱敏 | 值含 `password`/`pass`/`secret` 关键字的参数默认掩码，👁 切换显示 |
| 保存/另存 | 写回 YAML 文件 |
| 重新加载 | 放弃界面修改，从文件重读 |
| 校验引用 | 扫描当前勾选用例的 `{{group.param}}` 引用，核对环境配置是否都定义，列出未定义项 |
| 切换文件 | 下拉切换不同 env 文件（虽是单一全局配置，但可保存多份文件切换加载） |

### 7.3 与执行的联动

- 执行时若面板「启用环境配置」勾选 → 把当前加载的配置随引擎启动传入。
- 改动未保存时执行 → 提示「环境配置有未保存修改，是否保存后执行」。
- 环境配置面板的编辑不影响正在执行的用例（只读语义，执行用的是启动时加载的快照）。

## 9. 与其他模块的接口

| 接口 | 方向 | 说明 |
|------|------|------|
| 变量查找 | M7 → M2 | M2 模板替换的兜底层：用例级未命中 → 查环境配置（点号名 `{{group.param}}` 直接查环境配置） |
| 配置加载 | M7 → M3 | M3 `start` 时加载 env.yaml，注入用例模板替换上下文；加载失败 → 引擎 ERROR |
| CLI 选项 | M5 → M7 | `--env-config` 指定文件；`list env` 列出配置 |
| UI 编辑 | M6 → M7 | 环境配置面板可视化编辑、保存、校验引用、随执行传入 |
| 报告 | M7 → M4（预留） | 报告头可展示本次执行使用的环境配置快照（便于复现），后续迭代 |
