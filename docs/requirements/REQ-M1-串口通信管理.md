## 1. 模块概述

串口通信管理是整个测试工具的基础层，负责串口的配置、连接、数据收发、URC 处理和原始日志记录。所有上层模块（测试用例定义、执行引擎、报告等）均依赖本模块提供的串口通信能力。

本模块不依赖其他模块。

## 2. 串口参数配置

### 2.1 连接级参数

连接级参数在建立串口连接时确定，属于端口硬件通信参数，一旦设定在连接期间不变。

| 配置项 | 可选值 / 类型 | 默认值 | 说明 |
|--------|-------------|--------|------|
| 端口号 | 字符串（如 COM3、/dev/ttyUSB0） | 无（必填） | 串口设备路径 |
| 波特率 | 常用值 + 自定义数值 | 115200 | 支持常用值（1200~921600），也支持用户输入任意数值 |
| 数据位 | 5 / 6 / 7 / 8 | 8 | — |
| 停止位 | 1 / 1.5 / 2 | 1 | — |
| 校验位 | None / Even / Odd / Mark / Space | None | — |
| 流控 | None / RTS-CTS / XON-XOFF | None | — |
| 命令结束符 | `\r` / `\r\n` | `\r\n` | 发送命令时自动追加的结束符。仅这两种枚举值（AT 标准命令只有 CR / CRLF 两种结束符，不支持其他自定义字符串） |

### 2.2 行为级参数

行为级参数控制命令发送和响应等待的行为。

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| 响应超时 | 数值（秒） | 5（配置文件可改） | 单条命令发送后等待响应的超时时间。仅在**步骤级**配置（M2 步骤的 `timeout` 字段）；步骤未配时用配置文件默认值。无三级继承 |
| 发送间隔 | 数值（毫秒） | 0 | 两次命令发送之间的最小间隔时间。支持用例级/步骤级配置 |

**响应超时的配置规则（简化版）：**

- 超时只在**步骤级**配置，不存在用例级或全局级继承。
- 步骤显式配了 `timeout` → 用配置值。
- 步骤未配 → 用配置文件的默认值（默认 5 秒，可在配置文件修改）。
- 详细执行语义见 M3 §3。

**发送间隔的配置优先级（从高到低）：**

1. **步骤级**：用例中单个步骤指定的值，仅对该步骤生效
2. **用例级**：用例文件中声明的值，对该用例所有步骤生效
3. **默认值**：0（不等待）

### 2.3 配置管理

- 连接级参数支持运行时动态修改，修改后需重新建立连接生效
- 行为级参数支持运行时动态修改，立即生效
- 提供默认配置模板，用户可覆盖部分参数

## 3. 数据输入方式

串口发送数据支持两种输入方式：直接输入（短命令）、数据流输入。两种方式可在同一个测试用例的不同步骤中混合使用。

**设计原则：** 数据流输入不关心数据内部结构（文件格式、换行、编码等），统一按字节流发送，超阈值自动分块。不区分文件类型，不做逐行/注释/编码处理。

### 3.0 与 M2 用例字段对齐

本节定义的两种数据输入方式，对应 M2 用例步骤的两种输入字段（M2 §3.2 ~ §3.3）：

| M1 输入方式 | M2 步骤字段 | 参数对应 |
|------------|------------|---------|
| 直接输入（§3.1） | `command` | 自动追加结束符（§2.1） |
| 数据流输入（§3.2） | `data` | 参数命名一一对应（见下表） |

**数据流输入参数对应：** M1「数据来源/分块阈值/分块大小/块间间隔/追加结束符」→ M2 `data` 的 `file|inline/chunk_threshold/chunk_size/chunk_interval/append_terminator`。

本节其余内容为通信层行为定义，M2 用例字段是其用例层表达，两者参数语义完全一致。

### 3.1 直接输入（短命令）

适用于短文本命令（如 AT 指令），即当前已有的发送方式。

- 输入内容作为字符串直接发送
- 自动追加命令结束符（§2.1 中配置的 `命令结束符`）
- 受发送间隔（§2.2）约束

### 3.2 数据流输入

发送任意字节流：可来自文件或用例内联。工具不关心数据内部结构，统一按字节流发送；数据量超过阈值时自动分块传输。适用于固件升级、大报文发送、二进制数据流等场景。

**数据来源（二选一）：**

| 来源 | 说明 |
|------|------|
| 文件 | 从指定文件读取全部字节作为数据流发送 |
| 内联 | 在用例步骤中直接定义的文本内容，或从变量作用域引用的数据（如前序步骤提取的响应内容） |

> 文件以二进制方式读取，不解析内容、不转码、不处理换行。文件路径不存在时，当前步骤标记失败，记录错误日志。

**发送参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| 数据来源 | 文件 / 内联 | 无（必填） | 数据来源，二选一 |
| 分块阈值 | 数值（字节） | 4096 | 数据长度超过此值时自动启用分块发送；低于则整体发送 |
| 分块大小 | 数值（字节） | 1024 | 分块发送时每块的字节数 |
| 块间间隔 | 数值（毫秒） | 50 | 分块发送时两块数据之间的等待时间 |
| 追加结束符 | 布尔 | false | 数据发送完成后是否追加命令结束符 |

**发送行为：**

- **整体 vs 分块**：数据长度 ≤ 分块阈值时整体发送（一次性写入）；超过阈值时按分块大小分块，块间按块间间隔逐块写入串口。
- 若串口配置了流控（RTS-CTS / XON-XOFF），分块发送遵循流控信号。
- 发送过程中记录进度日志：分块发送时每块完成记录已发送字节数 / 总字节数。
- 发送完成后等待设备响应，响应超时遵循 §2.2 中配置的超时时间。

**发送中断处理（断连即失败）：**

- 数据发送期间串口断连，**当前步骤直接标记失败**，不重传、不续传。
- 断连触发 M1 §4.2 自动重连流程，重连期间正在执行的用例暂停等待；重连成功后由上层（M2）按失败处理策略（on_failure）决定后续动作，本模块不自行续传未发送完的数据。
- 理由：数据流无天然断点概念，续传会导致对端收到残缺数据，正确性无法保证。

**响应接收：**

- 数据发送后的响应可能分段返回，响应接收持续进行直到达到超时时间或收到完整响应标志（如 `OK` / `ERROR`）。
- 分段响应自动拼接为完整响应后返回给上层。响应完整性判定责任见 §7.5。

### 3.3 两种方式的对比与选择

| 维度 | 直接输入 | 数据流输入 |
|------|---------|-----------|
| 数据长度 | 短（通常 < 1KB） | 任意 |
| 数据来源 | 用例内联定义 | 文件 / 内联（含变量引用） |
| 适用场景 | AT 命令、短配置指令 | 固件升级、大报文、二进制数据流 |
| 分块支持 | 不需要 | 自动（超阈值） |
| 结束符处理 | 自动追加 | 可配置（默认不追加） |

## 4. 串口连接生命周期

### 4.1 连接管理

- **打开连接**：根据配置参数打开串口，建立通信通道
- **关闭连接**：释放串口资源，确保端口可被其他程序使用
- **连接状态查询**：随时可查询串口当前状态（已连接 / 已断开 / 重连中）

### 4.2 热插拔检测与自动重连

- 持续检测串口连接状态
- 检测到断开时自动触发重连流程

**重连策略：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 重连间隔 | 3 秒 | 每次重连尝试之间的等待时间 |
| 最大重试次数 | 10 次 | 超过后放弃重连 |

- 重连参数可配置
- 重连期间正在执行的步骤暂停等待（等待时间不计入步骤响应超时）
- 重连成功后，由上层（M3）从**当前用例**重新执行（用例级重试，不保存步骤级断点）
- 超过最大重试次数仍未成功，标记当前用例失败，继续执行下一个用例
- 若同一用例因断连连续重试达到安全阀值（如3次）仍未稳定，放弃该用例，继续下一个用例

### 4.3 操作取消（配合上层中断）

提供"取消当前阻塞操作"的能力，配合上层（M3）的 stop（中断当前用例）语义：

- 上层调用 stop 时，本模块立即取消当前端口正在阻塞的发送/接收操作。
- 阻塞的串口读操作立即返回（不等待响应超时），中断当前步骤。
- 取消后串口连接保持（不关闭），供后续用例继续使用。
- 详细中断语义见 M3 §7。

## 5. 多串口管理

### 5.1 多端口支持

- 单次工具运行可打开多个串口（执行参数指定），用于支持跨端口用例（双模块测试等场景）。
- 每个串口独立配置参数，独立记录日志。
- **执行模型为串行**（见 M3 §1/§6）：单次运行内用例串行执行，一个用例内的步骤也串行（含跨端口步骤）。同一时刻只有一个步骤在执行，端口间无并发竞争。

### 5.2 跨端口用例

跨端口联动场景（如客户端/服务端双模块测试）通过**同一用例内步骤分布在不同端口**实现：

- 用例的步骤通过 `port` 字段指定各自执行的串口（步骤级 port 覆盖用例级 port）。
- 步骤在指定端口串行执行，按顺序切换端口。
- **变量属于用例，不属于端口**：跨端口步骤共享同一用例变量池，前一步骤 extract 的变量后续步骤（含其他端口）立即可见。无跨端口变量池、无 export 机制、无竞态问题。

**URC 跨端口触发：**
- A 串口收到匹配 URC 规则的消息时，可触发 B 串口发送指定命令。
- 触发目标串口和命令在 URC 规则中指定。

### 5.3 多实例并行（外部并行）

单次运行内不并行。需要并行测试多端口时，**启动多个工具实例**，每个实例绑定自己的端口和用例，各自独立运行。实例间完全隔离（独立进程、端口、用例、结果）。

## 6. URC 处理

### 6.1 URC 监听

- 后台持续监听所有已连接串口的 URC
- URC 监听不影响正常命令的发送和响应接收

### 6.2 URC 匹配模式

支持四种匹配模式：

| 模式 | 说明 | 示例 |
|------|------|------|
| 前缀匹配 | URC 以指定前缀开头即匹配 | 前缀 `+CEREG:` 匹配 `+CEREG: 0,1` |
| 正则匹配 | URC 内容匹配指定正则表达式 | 正则 `\+CEREG:\s*\d,\d` |
| 包含匹配 | URC 包含指定字符串即匹配 | 包含 `CLOSED` |
| 精确匹配 | URC 完全等于指定内容 | 精确 `OK` |

### 6.3 URC 触发动作

URC 匹配成功后，支持以下触发动作：

| 动作 | 说明 |
|------|------|
| 记录日志 | 将 URC 内容记录到 URC 日志，带时间戳和串口标识 |
| 触发测试步骤 | 触发执行指定的测试用例或步骤 |
| 执行命令 | 在指定串口发送指定的 AT 命令 |

- 同一条 URC 可同时触发多个动作
- 触发动作中执行的命令同样受串口参数（超时、结束符等）约束

### 6.4 URC 与命令响应的区分

**基本策略：基于发送状态区分**

- **空闲状态**（未发送命令时）收到的数据归为 URC
- **等待响应状态**（发送命令后、未收到完整响应前）收到的数据归为命令响应

**增强策略：等待响应期间也提取 URC**

- 等待命令响应期间，同时对接收数据做 URC 前缀匹配
- 匹配到 URC 规则的数据同时归为 URC（触发相应动作），并从响应流中分离
- 确保在命令响应期间不遗漏任何 URC

### 6.5 URC 日志

- URC 日志独立于命令响应日志
- 每条 URC 日志记录：时间戳、串口标识、URC 原始内容、匹配的规则名称、触发的动作
- URC 日志按串口分组存储

## 7. 串口原始日志记录

### 7.1 记录内容

- 记录串口所有收发的原始数据
- 每条记录包含：方向标记（TX/RX）、时间戳、数据内容

### 7.2 记录格式

- 每个日志文件同时包含 HEX 和 TEXT 两种格式
- 每条数据的记录格式：

```
[2026-05-19 14:30:25.123] [TX] [TEXT] AT\r\n
[2026-05-19 14:30:25.123] [TX] [HEX]  41 54 0D 0A
[2026-05-19 14:30:25.456] [RX] [TEXT] OK\r\n
[2026-05-19 14:30:25.456] [RX] [HEX]  4F 4B 0D 0A
```

### 7.3 文件组织

- 按测试会话、端口、用例三个维度组织
- 每个用例每个端口生成一个独立的日志文件

目录结构：

```
logs/
└── 20260519_143025/          # 测试会话（开始时间）
    ├── COM3/                 # 端口
    │   ├── case_001.log      # 用例级别日志
    │   └── case_002.log
    └── COM5/
        ├── case_001.log
        └── case_002.log
```

### 7.4 日志保留

- 不自动清理日志文件
- 用户根据需要手动管理日志文件

### 7.5 响应完整性判定责任（与 M2 接口契约）

本模块（M1）负责判定命令响应是否完整，判定后将完整响应文本交付上层（M2）：

- **完整响应判定标志**：收到 `OK` / `ERROR` 等 AT 响应终结标志，或达到响应超时时间。
- 超时判定的响应视为"完整但超时"，连同超时标记一并交付。
- 响应完整性判定与原始日志记录（§7.1~§7.4）相互独立：日志记录所有收发字节流，完整性判定仅用于界定"一次命令响应"的边界。

**责任划分：**

- M1 负责界定响应边界，交付完整响应文本（及超时/异常标记）。
- M2 负责对完整响应做变量提取（extract）和断言（assert），不自行判定响应是否完整。
- 详细接口契约见 M2 §15。

## 8. 异常处理

### 8.1 异常类型与处理策略

| 异常类型 | 处理策略 |
|---------|---------|
| 串口断连 | 记录错误日志 → 触发自动重连 → 重连成功后由上层（M3）从当前用例重新执行（用例级重试） |
| 响应超时 | 记录错误日志 → 当前步骤标记失败 → 按步骤 on_failure 处理（见 M3 §4） |
| 数据错乱 | 记录错误日志（含原始数据）→ 当前步骤标记失败 → 按步骤 on_failure 处理 |
| 串口打开失败 | 记录错误日志 → 标记该端口不可用 → 用到该端口的用例标记失败，其他用例不受影响 |
| 重连超限 | 记录错误日志 → 当前用例标记失败 → 继续执行下一个用例 |
| 数据源文件不存在 | 记录错误日志（含文件路径）→ 当前步骤标记失败 → 按步骤 on_failure 处理 |
| 数据源文件读取失败 | 记录错误日志（含文件路径和错误原因）→ 当前步骤标记失败 → 按步骤 on_failure 处理 |
| 数据流发送中断 | 记录错误日志（含已发送字节数/总字节数）→ 当前步骤标记失败（**断连即失败，不续传**）→ 触发自动重连 → 重连成功后由上层（M3）从当前用例重新执行 |
| 分块发送超时 | 记录错误日志（含当前块号）→ 当前步骤标记失败 → 按步骤 on_failure 处理 |

### 8.2 资源管理

- 任何异常情况下确保正确释放串口资源
- 串口关闭时清理所有相关缓冲区
- 异常后重连成功前不发送新命令
