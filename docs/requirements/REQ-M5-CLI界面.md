# M5 CLI界面 — 需求文档

> 版本：v0.1（草稿）
> 日期：2026-06-19
> 状态：草稿
> 依赖：M2 测试用例定义（v1.4，§2 用例文件 / §12 套件 / §14 标签）、M3 测试执行引擎（v0.1，§7.2 start/stop / §7.4 进度事件 / §9 待细化项）、M4 测试报告（v0.1，§3 控制台输出 / §4 HTML 报告）

---

## 1. 模块概述

CLI 是工具的用户入口，职责是把命令行参数和配置文件翻译成 M3 引擎的 `start(配置)` 调用，订阅 M3 的进度事件并按 M4 约定的格式渲染控制台输出，执行结束后触发 M4 生成 HTML 报告。

**本模块只做"翻译与编排"，不产生测试逻辑：**

- 不解析用例语义（M2/M3 负责）。
- 不直接读写串口（M1 负责）。
- 不自行渲染报告内容（M4 负责）。

**核心设计原则：**

- **薄入口**：CLI 仅做参数解析、配置加载、事件订阅、控制台渲染、报告触发的编排，无业务逻辑。
- **配置兜底**：所有可变参数（端口、波特率、超时默认值等）都有配置文件默认值，命令行参数仅做覆盖。
- **第一阶段极简子命令**：只做 `run`、`list` 两个子命令；`report`（历史报告查看）延后，与 M4 §5.4 一致。

---

## 2. 子命令总览

| 子命令 | 作用 | 第一阶段 |
|--------|------|---------|
| `run` | 执行用例/套件，实时显示进度，结束生成报告 | ✅ |
| `list` | 列出可用用例、套件、串口 | ✅ |
| `report` | 查看/打开历史报告 | ❌（延后，M4 §5.4） |
| `version` | 显示版本号 | ✅ |

无子命令时等价于 `--help`，显示用法。

## 3. `run` 子命令

### 3.1 用法

```
atprobe run [用例或套件路径...] [选项]
```

- **位置参数**：一个或多个用例文件、套件文件、或目录路径。
  - 用例文件（如 `network-basic_register.yaml`）→ 直接执行。
  - 套件文件（如 `suite-network.yaml`）→ 执行套件内引用的用例。
  - 目录 → 执行该目录下所有用例文件（不含套件文件，避免重复）。
- 不提供路径 → 默认从配置文件的 `cases_dir` 加载全部用例。

### 3.2 选项

| 选项 | 说明 | 默认（配置文件兜底） |
|------|------|---------------------|
| `--port <表达式>` | 端口及串口参数，可重复，复合表达式（见 §3.3） | 配置文件 `ports` |
| `--baud <值>` | 默认波特率（未在 `--port` 表达式指定时使用） | 115200 |
| `--tag <标签>` | 标签过滤，可重复（见 §3.4） | 不过滤 |
| `--exclude-tag <标签>` | 排除标签，可重复 | 不排除 |
| `--config <文件>` | 指定配置文件路径 | `./atprobe.yaml` |
| `--env-config <文件>` | 指定测试环境配置文件（M7，跨用例共享的固化参数，服务器地址/版本号等） | `./env.yaml`（不存在则不加载，不报错） |
| `--no-color` | 关闭控制台颜色（ANSI） | 终端启用颜色、非终端自动关闭 |
| `--log-level <级别>` | 控制台日志详细度：`progress` / `debug` | `progress` |
| `--no-report` | 不生成 HTML 报告（仅控制台输出） | 生成报告 |
| `--report-dir <目录>` | 报告输出目录 | 配置文件 `report_dir`，默认 `./reports/` |
| `--dry-run` | 只解析与加载用例、检查端口可用性，不实际执行 | 关闭 |
| `--help` | 显示 run 用法 | — |

> 参数优先级（从高到低）：**命令行参数 > 配置文件 > 内置默认值**。

### 3.3 `--port` 复合表达式（跨端口用例核心）

跨端口用例（双模块测试）需同时打开多个端口，且各端口波特率可能不同。采用**复合表达式**，一条 `--port` 表达式描述一个端口的完整参数，可重复指定多个端口：

```
--port COM3                    # 仅端口名，其余用默认（baud=115200 等）
--port COM3:115200             # 端口名:波特率
--port COM3:115200:8N1         # 端口名:波特率:数据位校验位停止位
--port COM5:9600 --port COM3   # 重复指定，打开两个端口
```

**语法 BNF：**

```
port-expr  := PORTNAME [ ":" BAUD [ ":" FRAME ] ]
PORTNAME   := 串口名（COM3 / /dev/ttyUSB0 等，平台相关）
BAUD       := 数值（9600 / 115200 / 921600 等）
FRAME      := 数据位 校验位 停止位，紧凑写法，如 8N1 / 8E1 / 7O2
             数据位: 5/6/7/8
             校验位: N(无)/E(偶)/O(奇)
             停止位: 1/2
```

**与用例 `port` 字段的匹配规则：**

1. 命令行 `--port` 打开的端口形成"可用端口集"，用例/步骤的 `port` 字段从该集合中选取。
2. 用例未指定 `port` → 用可用端口集中的**第一个**作为默认端口。
3. 用例指定 `port: COM3` 但 `--port` 未提供 COM3 → **启动失败**（ERROR），提示端口未配置，不执行任何用例。
4. 不提供任何 `--port` → 使用配置文件 `ports` 列表；配置文件也没有 → **启动失败**。

> 复合表达式只解决"一次运行内打开多个端口"。并行测试多端口（多端口跑不同用例集）仍走多实例方式（M3 §2.3），每个实例配置自己的端口。

### 3.4 标签过滤（消费 M2 §14）

`--tag` 与 `--exclude-tag` 遵循 M2 §14 的标签规则：

- `--tag smoke` → 只执行带 `smoke` 标签的用例。
- `--tag smoke --tag network` → 多次 `--tag` 为**并集**（带任一标签即选中）。
- `--exclude-tag slow` → 排除带 `slow` 标签的用例。
- 过滤在套件和目录展开后的"最终用例列表"上执行。
- 标签过滤对套件内的用例同样生效（M2 §12.2 支持按标签筛选套件内部分用例）。

### 3.5 配置文件（消费 M3 §9 待细化项）

CLI 启动时加载配置文件（默认 `./atprobe.yaml`，可用 `--config` 覆盖）。配置文件定义所有可变参数的默认值，命令行参数覆盖之。

**配置文件结构（YAML）：**

```yaml
# atprobe.yaml
ports:                          # 默认端口列表（--port 未指定时使用）
  - COM3:115200:8N1
  - COM5:9600:8N1

default:
  step_timeout: 5               # 步骤级默认超时（秒），消费 M3 §5.1
  baud: 115200                  # --baud 未指定时的默认波特率
  log_level: progress           # 默认控制台日志级别

cases_dir: ./testcases          # 不提供用例路径时的默认加载目录
report_dir: ./reports           # HTML 报告输出目录
env_config: ./env.yaml          # 测试环境配置文件（M7），跨用例共享的固化参数；--env-config 覆盖

console:
  color: true                   # 默认启用颜色；非终端环境自动忽略（见 §6.3）
  command_truncate: 40          # 控制台命令显示截断长度（字符）

log:
  dir: ./logs                   # 原始日志根目录（透传给 M1）
  keep: 0                       # 日志保留份数，0=不自动清理（M1 §7.4）

pressure:                       # 压测默认值（消费 M3 §9 待定项之一）
  pass_rate_threshold: 95       # 压测用例 PASS 阈值（成功率%），低于则 FAIL
```

**配置加载与覆盖规则：**

1. 启动时读取配置文件，解析失败 → ERROR，提示行号与原因后退出。
2. 命令行参数对同名配置项**整体覆盖**（不是深度合并）。例外：`--port` 为累加（多个 `--port` 拼成端口集，不覆盖配置文件的 `ports`，而是替换之）。
3. 配置文件不存在 → 使用内置默认值，不报错（除非用户显式 `--config` 指定了路径却不存在 → 报错）。

> `pressure.pass_rate_threshold` 落地了 M3 §9 与 M4 §6 共同标注的"压测用例 PASS/FAIL 阈值"待定项。

### 3.6 `run` 执行编排

```
1. 解析命令行参数 + 加载配置文件（§3.5）
2. 解析位置参数（用例/套件/目录）→ 展开为最终用例列表
   ├─ 标签过滤（§3.4）
   └─ 用例加载失败（YAML 解析错误等）→ 报错列出，询问是否继续或中止
3. 打开 --port / 配置文件指定的端口（委托 M1）
   ├─ 任一端口打开失败 → 提示，按 M3 §7.5 场景C 处理
   └ --dry-run → 到此为止，打印将执行的用例清单和端口配置后退出
4. 调用 M3 engine.start(配置)   ← 配置含端口集、用例列表、超时默认、压测阈值等
5. 订阅 M3 进度事件（§6.2），按 M4 §3 格式渲染控制台实时输出
6. 期间监听 Ctrl+C（§5）
7. 引擎结束（FINISHED/ERROR）→ 调用 M4 生成 HTML 报告（除非 --no-report）
8. 控制台输出结果汇总（M4 §3.3）
9. 退出码：全 PASS=0，存在 FAIL/SKIPPED/INTERRUPTED=1，ERROR=2
```

## 4. `list` 子命令

### 4.1 用法

```
atprobe list [子目标] [选项]
```

### 4.2 子目标

| 子目标 | 作用 |
|--------|------|
| `cases`（默认） | 列出可用用例 |
| `suites` | 列出可用套件 |
| `ports` | 列出系统可用串口（委托 M1 枚举） |

### 4.3 `list cases` 输出

```
可用用例 (扫描目录: ./testcases):
  network/
    [network] 网络注册-基础验证          network-basic_register.yaml
    [network, p0] PDP激活               network-pdp_activate.yaml
  sms/
    [sms] 短信-发送文本                  sms-send_text.yaml
  共 3 个用例
```

- 支持 `--tag` 过滤（与 `run` 一致）。
- 显示用例名（M2 §12.3 的 name 字段）、标签、相对路径。

### 4.4 `list suites` 输出

```
可用套件:
  network/  网络注册测试套件          suite-network.yaml  (3 用例)
  sms/      短信测试套件              suite-sms.yaml      (2 用例)
  共 2 个套件
```

```
可用串口:
  COM3   (USB串口, 可用)
  COM5   (USB串口, 可用)
  COM7   (蓝牙串口, 占用中)
```

- 占用状态由 M1 检测（尝试独占打开）。
- 无子目标或参数时，`list` 默认等价于 `list cases`。

## 5. 中断信号处理（协调 M3 §7.2）

### 5.1 冲突与协调

M3 §7.2 定义引擎对外仅 `stop()`，且 `stop()` 只中断当前用例并继续后续；引擎层无"停止全部"能力。但 CLI 用户自然的 Ctrl+C 直觉是"全停"。

**协调方案：CLI 层把 Ctrl+C 升级为交互式提示，并对 M3 的 `stop()` 做最小扩充（仍属极简控制）。**

### 5.2 Ctrl+C 交互流程

```
[Ctrl+C 捕获] 执行中收到中断信号
  ↓
提示选择:
  [c] 中断当前用例，继续后续    → 调用 engine.stop(mode=current)
  [q] 停止全部，立即结束        → 调用 engine.stop(mode=all)
  [回车/其他] 忽略，继续运行    → 返回继续执行
  ↓
执行选择（若 5 秒无选择，默认按 [c] 处理）
```

### 5.3 对 M3 §7.2 的补充（一处变更，需回填 M3）

`stop()` 增加一个 `mode` 参数，仍保持"仅 start/stop 两个控制接口"的极简原则：

| 调用 | 语义 | 引擎状态 |
|------|------|---------|
| `engine.start(配置)` | 启动执行 | → RUNNING |
| `engine.stop(mode=current)` | 中断当前用例（INTERRUPTED），**继续后续** | 保持 RUNNING |
| `engine.stop(mode=all)` | 中断当前用例，**不再执行后续**，正常收尾 | → FINISHED |

> `stop(mode=all)` 不破坏"极简控制"——仍是同一个 stop 接口、仅多一个枚举参数。M3 §7.2 的核心语义（中断当前用例、步骤级立即中止、不执行 teardown、当前用例标 INTERRUPTED）在两种 mode 下完全一致，仅"是否继续后续用例"不同。

### 5.4 连续 Ctrl+C

- 第一次 Ctrl+C → 进入交互提示（§5.2）。
- 交互提示期间再按 Ctrl+C → 直接 `stop(mode=all)`（等同选 [q]，避免用户被困在提示里）。

## 6. 控制台输出（消费 M4 §3）

### 6.1 职责划分

控制台输出的**内容与格式由 M4 定义**（M4 §3），CLI 负责**订阅 M3 进度事件并驱动渲染**：

```
M3 进度事件  →  CLI（事件订阅 + 渲染调度）  →  M4 控制台格式（§3.2/§3.3）
```

- M4 §3.2 实时进度格式、§3.3 结果汇总格式、§3.4 颜色/截断原则 —— 直接采用，不重复定义。
- CLI 的职责是：把 M3 §7.4 的进度事件字段映射到 M4 §3.2 的输出行。

### 6.2 进度事件字段映射（消费 M3 §9 待细化项）

M3 §7.4 给出了进度事件骨架，M3 §9 标"进度事件详细字段依赖 M5"。此处落地：

| M3 进度事件 | 字段定义 | 映射到 M4 §3.2 输出行 |
|------------|---------|---------------------|
| 用例开始 | `case_name`, `case_index`, `total_cases`, `case_type`(regular/pressure) | `[HH:MM:SS] [i/n] 用例名` |
| 步骤开始 | `step_index`, `command`(截断前), `port`, `phase`(setup/steps/teardown) | （可选 debug 级输出） |
| 步骤结果 | `step_index`, `status`, `duration_ms`, `port`, `command`, `extracted_vars`, `error_msg`, `retry_count`, `poll_iterations` | `  → [端口] 命令 ...... 结果 (耗时)` |
| 压测进度 | `current_round`, `total_rounds`, `success`, `fail`, `avg_ms` | `  进度: i/n轮 | 成功 x | 失败 y | 平均 zms` |
| 用例结果 | `case_name`, `status`, `duration_ms`, `error_msg` | `用例结果: PASS/FAIL (耗时)` |
| 引擎结束 | `summary`(M3 §8.2 全字段) | 触发 M4 §3.3 汇总输出 |

**时间戳：** 输出行的 `[HH:MM:SS]` 由 CLI 在收到事件时用本地时间生成（事件本身不含时间戳，避免冗余）。

### 6.3 颜色控制（消费 M4 §6 待细化项）

| 来源 | 行为 |
|------|------|
| 终端（TTY） | 默认启用 ANSI 颜色，PASS 绿/FAIL 红/SKIPPED 黄/INTERRUPTED 灰 |
| 非终端（管道/重定向） | 自动关闭颜色 |
| `--no-color` | 强制关闭 |
| 配置 `console.color: false` | 强制关闭（命令行无 `--no-color` 时生效） |

优先级：`--no-color` > 配置 `console.color` > 自动检测。

### 6.4 `--log-level`

| 级别 | 输出范围 |
|------|---------|
| `progress`（默认） | M4 §3.2 实时进度 + §3.3 汇总（用例行 + 步骤结果行） |
| `debug` | 额外打印步骤开始事件、完整响应内容（不截断）、extract 变量值、retry/poll 细节 |

> 详情（断言对比、完整响应）始终在 HTML 报告中，控制台默认只给概要。

## 7. 报告触发

### 7.1 流程

```
引擎结束（FINISHED/ERROR）
  ↓
若 --no-report → 跳过报告生成
否则 → 调用 M4 生成 HTML 报告
  ├─ 报告目录: --report-dir 或配置 report_dir，默认 ./reports/
  ├─ 报告子目录: YYYYMMDD_HHMMSS（与日志会话目录一致，M4 §5.2）
  └─ 文件: report.html
  ↓
控制台输出报告路径（M4 §3.3 末尾）
```

### 7.2 报告与日志的目录关系

```
reports/20260618_143025/report.html     ← HTML 报告
logs/20260618_143025/                   ← 原始日志（M1 记录）
  ├── COM3/case_001.log
  └── COM5/case_002.log
```

- 报告目录与日志目录**同名同层**（同一时间戳），报告中用相对路径引用日志（M4 §5.1）。
- CLI 把 `log.dir` 配置透传给 M1，把 `report_dir` 传给 M4，时间戳由 CLI 在 start 时生成并共享给两者，保证目录名一致。

## 8. 内置变量 `{{timestamp}}`（消费 M3 §9 待细化项）

M3 §9 标"内置变量 timestamp 格式依赖 M5"。落地：

- `{{timestamp}}` 在用例模板替换时，取**该步骤执行时刻**的本地时间。
- 格式：`YYYY-MM-DD HH:MM:SS`（如 `2026-06-19 14:30:25`）。
- 仅这一个内置时间变量；不提供 `{{date}}`、`{{time}}` 等变体（避免膨胀，按需后续再加）。

> 其余内置变量（如循环计数 `{{loop_index}}`）归属 M2/M3，不在 M5 范围。

## 9. 退出码

| 退出码 | 含义 |
|--------|------|
| 0 | 全部用例 PASS |
| 1 | 存在 FAIL / SKIPPED / INTERRUPTED 用例 |
| 2 | 引擎 ERROR（端口全开失败、配置解析失败等致命错误，M3 §7.5 场景C） |
| 其他 | CLI 自身错误（参数解析失败、文件不存在等） |

> 退出码便于 CI/CD 集成（即使第一阶段不做 JUnit XML，退出码已能驱动流水线门禁）。

## 10. 典型使用场景

### 10.1 单端口基础测试

```
atprobe run testcases/network/network-basic_register.yaml --port COM3
```

### 10.2 标签过滤冒烟测试

```
atprobe run testcases/ --tag smoke --port COM3:115200
```

### 10.3 双模块跨端口测试（两个不同波特率端口）

```
atprobe run testcases/network/network-dual_module.yaml \
    --port COM3:115200 \
    --port COM5:9600
```

### 10.4 执行套件

```
atprobe run testcases/network/suite-network.yaml --port COM3 --tag regression
```

### 10.5 压测 + 仅控制台不生成报告

```
atprobe run testcases/stress/stress-loop_at.yaml --port COM3 --no-report
```

### 10.6 预演（不实际执行）

```
atprobe run testcases/ --port COM3 --dry-run
```

输出将执行的用例清单和端口配置，校验配置无误。

## 11. 待细化项

| 项 | 说明 | 依赖 |
|----|------|------|
| `list ports` 占用检测细节 | M1 端口枚举与占用判定的具体接口 | M1 |
| 配置文件 schema 校验 | 是否引入 JSON Schema 校验配置文件，还是仅靠解析报错 | 后续迭代 |
| 子命令补全（shell completion） | bash/zsh/powershell 补全脚本生成 | 后续迭代 |
| `report` 子命令 | 历史报告列出/打开/对比，与 M4 §5.4 联动 | 后续迭代 |

## 12. 与其他模块的接口

| 接口 | 方向 | 说明 |
|------|------|------|
| 用例加载 | M5 读 M2 | 加载并解析 M2 用例/套件 YAML，展开目录、应用标签过滤 |
