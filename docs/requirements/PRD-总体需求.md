## 1. 项目概述

### 1.1 项目背景

在嵌入式通信模组（蜂窝、WiFi、蓝牙等）的开发和测试过程中，需要频繁通过串口向设备发送AT命令并验证响应。手动测试效率低、容易遗漏、不可复现。本工具旨在实现AT命令测试的自动化，提升测试效率和覆盖率。

### 1.2 项目目标

- 提供一套通用的串口AT命令自动化测试工具
- 支持多种设备类型和多种测试场景
- 支持标准格式的测试用例定义，便于维护和复用
- 支持从AT指令集文档自动生成测试用例
- 提供清晰的测试报告

### 1.3 目标用户

- 嵌入式软件测试工程师
- 模组FAE（现场应用工程师）
- 研发工程师（自测验证）

## 2. 目标设备

工具需支持多种类型的串口设备，不限定于特定厂商或型号：

| 设备类型 | 典型代表 | AT命令特点 |
|---------|---------|-----------|
| 蜂窝模组（4G/5G/NB-IoT） | 移远、芯讯通、广和通、中兴等 | 命令集丰富，涉及网络注册、数据拨号、短信、电话等 |
| WiFi/蓝牙模组 | ESP32、乐鑫等 | 命令相对简单，涉及连接、数据收发等 |
| 通用串口设备 | 不限定 | 需灵活适配，不假设特定命令集 |

## 3. 测试场景

### 3.1 基础功能验证

**描述：** 发送单条AT命令，验证响应内容是否符合预期。

**功能要求：**

- 发送AT命令并等待响应
- 支持多种断言方式：
  - 响应包含指定字符串
  - 响应匹配正则表达式
  - 响应等于指定内容
  - 响应不包含指定内容
- 支持设置响应超时时间
- 支持配置命令结束符：`\r` 或 `\r\n`（默认 `\r\n`）
- 记录命令、响应内容、耗时、通过/失败状态

**示例：**

```
发送: AT
期望: 包含 "OK"
超时: 5秒
```

### 3.2 命令序列测试

**描述：** 按顺序执行多条AT命令，后续命令可依赖前序命令的结果。

**功能要求：**

- 按顺序执行命令列表
- 支持从响应中提取变量，在后续命令中使用
- 支持条件分支：根据前一步结果决定执行路径
- 支持命令间设置等待/延时
- 单步失败时可选择：中止序列 / 跳过继续 / 标记失败继续

**示例流程：**

```
步骤1: AT → 期望 OK
步骤2: AT+CPIN? → 提取状态 → 若 READY 则继续
步骤3: AT+CEREG? → 提取注册状态
步骤4: AT+CGDCONT=1,"IP","cmnet" → 期望 OK
步骤5: AT+CGACT=1,1 → 期望 OK
```

### 3.3 性能压测

**描述：** 循环执行命令或命令序列，统计响应时间、成功率等指标。

**功能要求：**

- 支持单命令循环压测
- 支持命令序列循环压测：将一组命令序列作为整体循环执行，每轮执行完整的命令序列
- 指定循环次数或执行时长
- 统计每次响应时间（最小值、最大值、平均值、P95、P99）
- 统计成功次数、失败次数、成功率、失败率
- 命令序列压测时，按单步和整体两个维度统计成功/失败
- 支持设置循环间隔
- 支持多串口并行压测
- 结果以表格和图表形式展示在报告中

**示例1：单命令压测**

```
命令: AT
循环次数: 100
间隔: 100ms
统计: 响应时间分布、成功/失败次数、成功率
```

**示例2：命令序列压测**

```
序列:
  步骤1: AT → 期望 OK
  步骤2: AT+CPIN? → 期望 READY
  步骤3: AT+CEREG? → 期望 注册成功
  步骤4: AT+CSQ → 期望 有效信号值
循环次数: 50
间隔: 500ms
统计: 每步成功/失败次数、整体序列成功率、响应时间分布
```

## 4. 串口通信

### 4.1 串口配置

**功能要求：**

- 支持配置串口参数：端口号、波特率、数据位、停止位、校验位、流控
- 支持的默认波特率：9600, 19200, 38400, 57600, 115200, 921600 等
- 支持运行时动态修改串口配置
- 支持串口热插拔检测和自动重连

### 4.2 多串口并行

**功能要求：**

- 支持 2-4 个串口同时工作
- 每个串口独立配置、独立执行、独立记录
- 各串口测试互不干扰
- 报告中按端口分组展示结果

### 4.3 URC（主动上报）处理

**描述：** 模块会主动上报消息（URC），不依赖主机命令。工具需监听、记录并支持基于URC触发动作。

**功能要求：**

- 后台持续监听URC
- 支持URC规则定义：匹配模式（前缀匹配、正则匹配）
- 支持URC触发动作：
  - 记录日志
  - 设置变量
  - 触发指定测试步骤
  - 执行指定命令
  - 发送告警通知
- URC日志独立记录，与命令响区分
- 不影响正常命令发送和响应接收

**典型URC示例：**

| URC前缀 | 含义 | 可能的触发动作 |
|---------|------|--------------|
| `+CEREG:` | 网络注册状态变化 | 记录状态变更 |
| `+CMT:` | 收到短信 | 触发短信读取流程 |
| `+CGEV:` | PDP上下文变化 | 触发重连流程 |
| `CLOSED` | 连接断开 | 触发重连或告警 |
| `CONNECT` | 数据连接建立 | 记录连接事件 |
| `+CUSD:` | USSD响应 | 记录USSD内容 |

### 4.4 串口原始日志记录

**描述：** 记录串口所有收发的原始数据，用于异常和失败用例的事后分析。

**功能要求：**

- 同时记录串口所有收发数据，包括发送的命令和接收的响应
- 支持两种记录格式：
  - **HEX格式**：原始字节数据以十六进制形式记录，用于排查非可见字符、编码等问题
  - **TEXT格式**：可读文本形式记录，便于快速浏览和人工分析
- 日志中标记每条数据的方向（RX 接收 / TX 发送）和时间戳
- 日志按用例、端口、时间三个维度独立存储，目录结构示例：

```
logs/
└── 20260519_143025/          # 测试会话（时间）
    ├── COM3/                 # 端口
    │   ├── case_001.yaml.log # 用例级别日志
    │   └── case_002.yaml.log
    └── COM5/
        ├── case_001.yaml.log
        └── case_002.yaml.log
```

- 每个日志文件同时包含HEX和TEXT两种格式的内容
- 失败用例在报告中可直接关联到对应的原始日志文件，便于快速定位

## 5. 测试用例

### 5.1 用例格式

**统一使用YAML作为唯一用例定义格式。**

- AT命令测试本质是声明式的（"发什么、期望什么"），YAML天然匹配
- 统一格式降低学习成本，用户只需掌握一种语法
- AI生成用例目标格式统一，简化生成流程
- 具体的YAML Schema设计详见独立设计文档

### 5.2 用例组织

- 支持用例套件（Test Suite）：多个相关用例的集合
- 支持标签（Tag）：按功能模块、优先级等分类
- 支持跨端口变量共享
- 支持前置条件检查，条件不满足时跳过

## 6. 测试执行与调度

### 6.1 执行控制

**功能要求：**

- 支持执行单个用例、用例套件、全部用例
- 支持按标签筛选用例执行
- 支持指定串口执行
- 支持执行前的前置条件检查
- 支持超时控制：单步超时、用例超时、全局超时

### 6.2 失败处理

**功能要求：**

- 记录失败用例的详细信息：命令、期望、实际响应、失败原因
- 失败后继续执行后续用例（不影响其他用例）
- 单个用例内的步骤失败策略可配置：中止 / 跳过 / 继续
- 汇总失败统计：失败数、失败率、失败用例列表

### 6.3 测试上下文

**功能要求：**

- 变量作用域为**用例级**：变量属于当前用例，用例之间独立，不跨用例传递（详见 M2 §5.3）
- 跨端口联动（如双模块测试）通过同一用例内步骤分布在不同端口实现，共享该用例变量池
- 支持从命令响应中提取值并赋给变量（extract）
- 支持变量在命令和断言中引用
- 支持内置变量：时间戳、串口号、循环计数等
- **测试环境配置（M7）**：提供跨用例共享的全局只读配置层，承载相对固化的测试参数（FTP/HTTP/TCP/MQTT 服务器地址端口、FOTA 版本号与升级包名、设备型号等）。用例通过 `{{group.param}}` 点号引用，执行时自动填充到 AT 指令中。配置可扩展（用户可任意增组增参数），支持文件编辑与 UI 编辑。详见 M7

## 7. 报告

### 7.1 报告内容

**功能要求：**

- 测试概览：总用例数、通过数、失败数、跳过数、通过率
- 执行耗时：总耗时、单用例耗时
- 用例详情：每个用例的步骤、命令、响应、断言结果
- 失败详情：失败原因、实际响应与期望的对比
- 性能统计：响应时间分布（压测场景）
- URC日志：测试期间的URC记录

### 7.2 报告格式

- **HTML报告**：可视化展示，支持浏览器查看（主要格式）
- **控制台输出**：实时显示执行进度和结果
- **JUnit XML**：用于CI/CD集成（预留）

### 7.3 报告管理

- 报告按时间戳命名存储
- 支持历史报告查看和对比（预留）

## 8. 用户界面

### 8.1 第一阶段：命令行（CLI）

**功能要求：**

- 提供命令行入口，支持子命令：
  - `run`：执行测试用例/套件
  - `list`：列出可用用例/串口
  - `report`：查看历史报告
- 支持参数配置：串口号、波特率、用例路径、标签过滤等
- 实时显示执行进度
- 支持通过配置文件设置默认参数

### 8.2 第二阶段：桌面端 GUI（详见 REQ-M6）

> 修订（2026-06-19）：原标注「Web界面（预留）」经需求方确认调整为**桌面端 GUI 应用**（类似 SSCOM、友善串口助手等常见串口工具），非 Web。远程执行、多人协作不实现。详见 `REQ-M6-UI管理界面.md`。

**第一阶段功能范围：**

- 端口管理与配置（端口选择、参数配置、连接/断开、多端口）
- 快捷发送 AT 指令（手动调试模式，类似串口助手手动发送）—— PRD 原未列，需求方新增
- 用例筛选与执行（加载用例、勾选/标签筛选/搜索、驱动引擎执行）
- 实时监控（端口收发数据流 + 用例执行进度与结果）
- 报告查看（内嵌 WebView 打开 M4 生成的 HTML 报告）

**延后/不实现：**

- 用例可视化编辑（第一阶段仅外部编辑 YAML）
- 报告对比
- 远程执行测试（不实现）
- Web 界面（调整为桌面端 GUI）

## 9. 非功能性需求

### 9.1 可靠性

- 串口通信异常时（断连、超时）不崩溃，记录错误并继续
- 异常情况下正确释放串口资源
- 大量用例执行时内存稳定，无泄漏

### 9.2 可扩展性

- 架构分层清晰，各层通过接口交互
- 新增报告格式只需实现报告接口
- 新增设备类型只需配置适配

### 9.3 易用性

- 零代码基础可通过YAML编写用例
- CLI命令简洁直观
- 提供丰富的示例用例
- 错误提示清晰，便于排查问题

### 9.4 性能

- 命令发送和响应接收延迟 < 10ms（不含串口本身延迟）
- 支持同时管理4个串口不卡顿
- 单个用例执行开销可忽略

## 10. 约束与假设

### 10.1 约束

- 运行环境：Windows / Linux / macOS
- 需要物理串口或USB转串口设备
- 串口通信基于文本模式（AT命令标准）

### 10.2 假设

- 目标设备支持标准AT命令格式（`AT` 前缀，`\r\n` 结束符）
- 设备响应格式为文本
- 测试人员对AT命令有基本了解

## 11. 术语表

| 术语 | 说明 |
|------|------|
| AT命令 | Hayes标准命令集，以 `AT` 开头的文本指令 |

| Test Suite | 测试套件，一组相关测试用例的集合 |
| Test Case | 测试用例，单个测试场景的定义 |

## 附录A：典型测试流程

```
1. 配置串口参数（端口、波特率等）
2. 打开串口连接
3. 启动URC监听
4. 加载测试用例/套件
5. 按顺序执行用例：
   a. 检查前置条件
   b. 逐步执行命令
   c. 接收响应并断言
   d. 提取变量（如需要）
   e. 记录结果
6. 执行后置清理
7. 生成测试报告
8. 关闭串口连接
```
