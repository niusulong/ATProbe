## 1. 模块概述

M6 是 ATProbe 的**桌面端图形界面（GUI）**，定位为类似 SSCOM、友善串口助手等常见桌面端串口工具的可视化操作入口，覆盖日常调试与测试场景。

### 1.1 与 PRD 的偏离（需记录）

PRD §8.2 原标注「第二阶段：Web界面（预留）」，含「远程执行测试」。经需求方确认，M6 实际定位调整为：

| PRD 原设想 | 实际调整 |
|-----------|---------|
| Web 界面（浏览器访问） | **桌面端 GUI 应用**（非 Web） |
| 远程执行测试 | **不实现**（本地工具） |
| 多人协作 | **不实现**（单用户本地工具） |

> 本文件取代 PRD §8.2 的原描述。PRD 与 README 中的 M6 表述需同步更新（见 §11）。

### 1.2 核心定位

- **桌面端单用户工具**：本地运行，通过操作系统串口 API 直接操作硬件，不经网络。
- **与 CLI 平级**：M6 与 M5 CLI 是**同一引擎的两个入口**，共享 M1-M4 全部能力，互不依赖。
- **交互式调试优先**：除「用例执行」外，M6 还提供「快捷发送 AT 指令」的手动调试模式——这是 PRD 未列、需求方明确要求的核心场景，类似串口助手的手动发送功能。

### 1.3 核心设计原则

- **进程内复用**：M6 与 M3 引擎在同一进程内，直接调用 `engine.start/stop`，不经 HTTP/网络。M3 **无需**封装为 HTTP API、**无需**任务队列。
- **两种执行模式并存**：
  - **用例执行模式**：加载 M2 用例，驱动 M3 引擎执行，复用 M4 报告。
  - **手动调试模式**：用户在发送框直接输入 AT 指令，**直接调用 M1**（不经 M3 引擎、不产生用例结果），用于探索性调试。
- **实时可见**：端口收发数据、用例执行进度、AT 指令响应均实时刷新展示。
- **只读报告 + 实时监控**：报告查看复用 M4 生成的 HTML（内嵌 WebView 打开）；执行过程实时监控在主界面内完成。
- **选项卡化 + 可扩展**：主工作区采用多文档选项卡（§2.3），功能以「选项卡类型」形式插接，新增功能不改主框架。界面现代化、美观（§2.2）。
- **现代化美观**：扁平化设计、浅/深主题、矢量图标、等宽字体显示收发数据，达到主流桌面应用视觉水准。

### 1.4 第一阶段功能范围

| 功能 | 说明 | 第一阶段 |
|------|------|---------|
| 端口管理与配置 | 端口选择、参数配置、连接/断开 | ✅ |
| 快捷发送 AT 指令 | 手动调试模式，发送框 + 响应区 | ✅ |
| 用例筛选与执行 | 加载用例、筛选、勾选、驱动引擎执行 | ✅ |
| 实时监控 | 端口收发流、用例执行进度与结果 | ✅ |
| 报告查看 | 执行结束后查看 HTML 报告 | ✅ |
| 用例可视化编辑 | YAML 编辑器（文本编辑 + 语法校验） | ❌（延后） |
| 报告对比 | 多次执行结果对比 | ❌（延后） |
| 远程执行 | — | ❌（不实现） |

## 2. 主界面布局

### 2.1 整体结构与设计风格

桌面端采用「侧边导航 + 多文档选项卡工作区」布局，三栏式：

```
┌─────────────────────────────────────────────────────────────────────┐
│ 标题栏：ATProbe                              [─ □ ×]               │
├─────────┬───────────────────────────────────────────┬───────────────┤
│         │  选项卡栏：[× COM3-调试] [× 用例执行] [× COM3监控] [+]    │
│         ├───────────────────────────────────────────┤               │
│  侧边栏  │                                           │               │
│ (导航)   │                                           │   右侧栏      │
│         │       当前选项卡内容（动态）                │ (上下文相关)  │
│ ▼ 端口   │                                           │               │
│ ▼ 用例   │                                           │               │
│ ▼ 报告   │                                           │               │
│ ▼ 设置   │                                           │               │
├─────────┴───────────────────────────────────────────┴───────────────┤
│ 状态栏：●COM3 已连接 · 引擎 IDLE · 就绪                       14:30 │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 设计风格（现代化要求）

界面须达到现代桌面应用的视觉与交互水准：

| 维度 | 要求 |
|------|------|
| **视觉风格** | 扁平化 + 现代化，圆角卡片、柔和阴影、充足留白；避免传统工具软件的密集控件堆砌 |
| **配色** | 支持浅色/深色主题切换（跟随系统或手动）；语义色统一（PASS 绿/FAIL 红/SKIPPED 黄/INTERRUPTED 灰，与 M4 一致） |
| **图标** | 矢量图标库（如 Material/Lucide 风格），支持主题色继承，清晰可辨 |
| **字体** | 等宽字体用于收发数据/AT 指令（如 JetBrains Mono / Cascadia Code）；比例字体用于 UI 文案 |
| **动效** | 克制而流畅：选项卡切换、状态变化、进度更新有过渡动画，不浮夸、不干扰阅读 |
| **间距与层级** | 统一的间距规范（4/8/16px 栅格），视觉层级通过字号、字重、颜色、间距体现，非边框堆叠 |
| **响应式** | 窗口可缩放，三栏比例可调且记忆；窄屏时侧边栏可折叠为图标条 |

> 配色/间距/字体的具体规范在 TSD-M6 技术选型时定义设计令牌（design token）。

### 2.3 选项卡系统（核心扩展机制）

**主工作区采用多文档选项卡（MDI Tabs）**，而非固定数量的预设面板。这是支撑后续功能扩展的核心架构决策。

**选项卡特性：**

- **可新增**：用户可随时新建选项卡，选择类型（见下表）。`[+]` 按钮或侧边栏双击触发。
- **可关闭**：每个选项卡带 `[×]`，关闭即销毁该选项卡实例（执行中的选项卡关闭前提示确认）。
- **可多开同类型**：如同时开 3 个「手动调试」选项卡分别绑 COM3/COM5/COM7，互不干扰。
- **可拖拽重排**：选项卡顺序可拖拽调整。
- **状态持久化**：打开的选项卡集合（类型 + 绑定参数）随 GUI 配置保存，重启恢复。
- **激活态记忆**：上次激活的选项卡重启后自动激活。

**选项卡类型注册表（插件化，支撑后续扩展）：**

| 类型 | 绑定 | 第一阶段 | 说明 |
|------|------|---------|------|
| `manual_debug` | 单端口 | ✅ | 手动调试（§4），可多开，每个绑不同端口 |
| `case_execute` | 用例选择集 | ✅ | 用例执行（§5），通常单开 |
| `monitor` | 端口集 | ✅ | 端口数据流监控（§6.2），可多开不同端口组合 |
| `execution_progress` | 执行实例 | ✅ | 用例执行进度（§6.3），执行时自动弹出 |
| `report_view` | 报告文件 | ✅ | 报告查看（§7），WebView 内嵌 |
| `env_config` | 环境配置文件 | ✅ | 测试环境配置编辑（M7），分组编辑/增删组参数/校验引用 |
| `case_editor` | 用例文件 | ❌延后 | 用例可视化编辑（YAML 编辑器） |
| `report_compare` | 报告对 | ❌延后 | 报告对比 |
| *自定义* | — | 扩展 | 后续新增功能按此注册表扩展，无需改布局架构 |

> 新增功能 = 注册一个新选项卡类型 + 实现其内容视图，无需改动主框架。侧边栏、状态栏、选项卡栏作为稳定外壳。

### 2.4 三栏职责

| 区域 | 职责 |
|------|------|
| **侧边栏（导航）** | 折叠式导航：端口（列表 + 连接状态）、用例（树 + 勾选）、报告（历史列表）、环境配置（M7，编辑/校验）、设置（主题/参数）。点击项可新建对应选项卡 |
| **主工作区（选项卡栏）** | 多文档选项卡容器，承载各类功能视图（§2.3） |
| **右侧栏** | 上下文相关，随当前激活选项卡切换内容：手动调试→响应区/历史；执行→结果/日志/URC；可隐藏 |
| **状态栏** | 全局状态汇总（端口连接数、引擎 RUNNING/IDLE、当前进度、系统时间） |

## 3. 端口管理与配置（左栏 - 端口面板）

### 3.1 功能

复用 M1 §2（参数配置）、§4.1（连接管理）、§5（多串口管理）能力，GUI 化呈现：

- **端口列表**：委托 M1 枚举系统串口（对应 CLI `list ports`），显示名称与占用状态。
- **参数配置**：下拉框/输入框配置 M1 §2.1 连接级参数（端口号、波特率、数据位、停止位、校验位、流控）+ §2.1 行为级参数（命令结束符、响应超时、发送间隔）。
- **连接/断开**：点击按钮打开/关闭端口（委托 M1 §4.1），状态实时反馈。
- **多端口**：支持同时管理多个端口（M1 §5.1），跨端口用例所需端口需先在此连接。

### 3.2 端口面板布局

```
┌─ 端口 ──────────────────────────┐
│ [刷新] [+ 新建连接]             │
├─────────────────────────────────┤
│ ● COM3  115200 8N1  [已连接]    │
│   [配置] [断开]                 │
│                                 │
│ ○ COM5  9600 8N1    [未连接]    │
│   [配置] [连接]                 │
└─────────────────────────────────┘
```

- 圆点标识连接状态（●绿=已连接，○灰=未连接，红=异常）。
- 每个端口一行，展开可改参数。
- 连接失败时行内显示错误原因（如「端口被占用」「波特率不支持」）。

### 3.3 配置变更生效规则（遵循 M1 §2.3）

| 参数类型 | 运行时修改 |
|---------|-----------|
| 连接级（波特率/数据位等） | 需断开重连生效（M1 §2.3）；GUI 弹确认提示 |
| 行为级（超时/间隔/结束符） | 立即生效（M1 §2.3） |

### 3.4 配置持久化

- 端口配置可保存为预设（命名，如「模组A-115200」），下次启动加载。
- 预设存储位置与 M5 配置文件（`atprobe.yaml`）解耦，存于 GUI 独立配置（如 `%APPDATA%/atprobe/ui-presets.yaml`），避免污染 CLI 配置。

## 4. 快捷发送 AT 指令（主工作区 - 手动调试 Tab）

### 4.1 定位

**这是 M6 区别于 CLI 的核心价值之一**——提供类似串口助手的手动调试能力。用户无需编写用例即可快速验证设备响应。

### 4.2 工作模式

手动调试模式**直接调用 M1**，不经 M3 引擎：

- 用户选定一个已连接端口（左栏端口面板）。
- 在发送框输入 AT 指令（或数据）。
- 调用 M1 §3.1（直接输入）或 §3.2（数据流输入）发送，接收响应展示。
- **不产生用例结果、不进 ExecutionResult、不生成报告**——纯调试行为。
- 收发数据仍记录原始日志（M1 §7），便于事后回看。

### 4.3 手动调试区布局

```
┌─ 手动调试  端口: [COM3 ▼] ──────────────────────────────────────┐
│                                                                 │
│  发送:                                                          │
│  ┌─────────────────────────────────────────────┐ [发送] [清空]  │
│  │ AT+CSQ                                       │               │
│  └─────────────────────────────────────────────┘               │
│  ☐ 追加结束符[\r\n ▼]  超时[5]秒   发送间隔[0]ms  ☐ HEX显示     │
│                                                                 │
│  响应:                                                          │
│  ┌─────────────────────────────────────────────┐ [保存] [清空]  │
│  │ [14:30:25] TX> AT+CSQ\r\n                    │               │
│  │ [14:30:25] RX< +CSQ: 23,0\r\n                │               │
│  │ [14:30:25] RX< OK\r\n                        │               │
│  └─────────────────────────────────────────────┘               │
│                                                                 │
│  快捷指令: [AT] [AT+CSQ] [AT+CEREG?] [AT+CPIN?] [+] [管理]     │
│  历史指令: [▼ AT+CSQ  AT+CPIN?  AT  ...]                       │
└─────────────────────────────────────────────────────────────────┘
```

### 4.4 功能要点

| 功能 | 说明 |
|------|------|
| **端口选择** | 下拉选已连接端口（左栏已连接的）；未连接则禁用发送 |
| **发送框** | 支持单行/多行；多行按行依次发送（行间遵守发送间隔） |
| **发送选项** | 结束符、超时、间隔与 M1 §2 行为级参数一致；HEX 显示开关切换响应区的显示格式 |
| **响应区** | 实时显示收发数据（带时间戳、TX/RX 方向标记），对应 M1 §7.2 原始日志格式 |
| **快捷指令** | 用户自定义常用 AT 指令按钮，一键发送；可增删管理（持久化） |
| **历史指令** | 自动记录最近发送的指令（下拉回选），避免重复输入 |
| **HEX 显示** | 响应区可切换 HEX/TEXT 视图（对应 M1 §7.2 的两种格式） |
| **多行/数据流** | 切换到数据流模式可发送文件内容（M1 §3.2） |

### 4.5 手动调试与用例执行的隔离

- 手动调试占用的端口，若同时被用例执行使用，需提示冲突。
- **建议规则**：同一端口同时只能被一种模式占用（手动调试 OR 用例执行），避免收发数据混杂。
- 手动调试发送的数据和用例执行的步骤结果在日志中通过「会话类型」标记区分。

## 5. 用例筛选与执行（主工作区 - 用例执行 Tab）

### 5.1 用例加载与展示

- 从左栏「用例树」加载用例/套件/目录（复用 M2 §12 目录结构、M5 §4 加载逻辑）。
- 用例树按功能域目录组织，层级：`功能域 → 套件/用例文件 → 用例`。
- 每个用例节点显示：名称（M2 §12.3 name 字段）、标签、来源文件。

### 5.2 用例树布局

```
┌─ 用例 ──────────────────────────────────┐
│ [刷新] [筛选▼] [全选] [全不选]           │
│ 搜索: [........................] 🔍     │
├─────────────────────────────────────────┤
│ ▼ 📁 testcases/network                  │
│   ▼ 📄 suite-network (套件)             │
│     ☑ 网络注册-基础验证 [network,p0]    │
│     ☑ PDP激活 [network]                 │
│     ☐ 网络注册-深度测试 [network,slow]  │
│   ☑ network-basic_register.yaml         │
│ ▼ 📁 testcases/sms                      │
│   ☐ 短信-发送文本 [sms]                 │
└─────────────────────────────────────────┘
```

### 5.3 筛选能力（需求方重点要求）

| 筛选方式 | 说明 |
|---------|------|
| **勾选** | 树节点复选框，逐个勾选要执行的用例；支持套件级勾选（勾选套件 = 选中套件全部用例） |
| **标签筛选** | 下拉多选标签（从所有用例 tags 聚合），选中后树中仅高亮匹配用例（对应 CLI `--tag`） |
| **文本搜索** | 关键词模糊匹配用例名/文件名，实时过滤树 |
| **全选/全不选** | 当前过滤结果集上的批量操作 |

> 勾选是最终执行依据：执行时只跑被勾选的用例，与筛选高亮无关（筛选只辅助查找）。此规则需在 UI 上明确提示。

### 5.4 执行参数配置

执行前在用例执行 Tab 顶部配置：

```
┌─ 用例执行 ───────────────────────────────────────────────────┐
│ 参与端口: ☑COM3 ☑COM5 ☐COM7   默认端口:[COM3 ▼]            │
│ 标签过滤: [network ▼]   排除:[slow ▼]   生成报告: ☑         │
│ 压测阈值: [95]%         报告目录:[./reports ▼]               │
│                                                              │
│ [▶ 执行选中]  [⏸ 停止]  [预演(Dry Run)]  [清空结果]         │
└──────────────────────────────────────────────────────────────┘
```

- **参与端口**：勾选左栏已连接端口（跨端口用例需多选）；对应 CLI `--port`。
- **默认端口**：用例未指定 port 时使用的端口；对应 M3 §6.2 默认端口。
- **压测阈值**：对应 M5 §3.5 `pressure.pass_rate_threshold`，GUI 可调。
- 其余选项与 CLI 选项一一对应（报告开关、报告目录、dry-run）。

### 5.5 执行控制

| 按钮 | 对应引擎动作 |
|------|-------------|
| **▶ 执行选中** | 调用 `engine.start(配置)`，配置含勾选用例 + 参与端口 + 阈值 |
| **⏸ 停止** | 弹出对话框：中断当前用例继续后续 / 停止全部（对应 `stop(mode=current\|all)`，M3 §7.2 / M5 §5.3） |
| **预演** | `--dry-run`，校验用例和端口配置，不实际执行 |
| **清空结果** | 清空右栏执行结果，不影响端口与用例勾选 |

### 5.6 端口冲突检查

- 执行前校验：所有参与端口必须处于「已连接」状态。
- 执行期间端口被引擎占用，左栏端口面板对应端口显示「执行中」标识，禁止手动调试占用。
- 执行结束后端口释放，恢复可用。

## 6. 实时监控（主工作区 - 实时监控 Tab + 右栏）

### 6.1 双层实时性

M6 的实时监控分两个层次，对应不同的数据源：

| 层次 | 数据源 | 内容 | 更新频率 |
|------|-------|------|---------|
| **端口数据流** | M1（直接） | 原始收发字节流（TX/RX），与手动调试响应区同源 | 实时（字节级） |
| **用例执行进度** | M3 进度事件 | 用例/步骤状态变化，对应 M3 §7.4 | 事件驱动 |

### 6.2 端口数据流监控（独立于手动调试）

实时监控 Tab 可**独立观察**端口数据流，即使该端口正在被用例执行使用：

```

│ [COM3] 14:30:25 RX< +CSQ: 23,0\r\n                          │
```

- 多端口数据合并显示时加端口前缀 `[COMx]` 区分。
- HEX/TEXT 切换（M1 §7.2）。
- 支持导出当前监控段为文件（复用 M1 §7.3 日志格式）。

### 6.3 用例执行进度监控（消费 M3 §7.4）

### 6.4 URC 监控面板

- 右栏可切换到 URC 面板，实时显示 M1 §6 捕获的 URC（带时间戳、来源端口、匹配规则）。
- URC 与命令响应分栏展示（M1 §6.4 逻辑隔离）。

## 7. 报告查看

### 7.1 实现方式

- 执行结束后，M4 生成 HTML 报告（M4 §4），路径在状态栏/右栏提示。
- M6 用**内嵌 WebView**（或调用系统默认浏览器）打开 `report.html`。
- 报告内容完全复用 M4 §4 的纯静态 HTML，M6 不重新渲染。

### 7.2 历史报告列表

- 菜单「报告 → 历史报告」：列出 `report_dir`（M5 §3.5）下的所有报告，按时间倒序。
- 每条显示：时间戳、用例总数/通过/失败（从报告头解析）、大小。
- 双击打开查看。
- **报告对比**：延后（与 M4 §6 待细化项一致）。

### 7.3 报告与执行的关联

- 用例执行结束后，右栏结果区顶部显示「查看报告」按钮，直接打开本次报告。
- 报告路径与原始日志路径（M1 §7.3）都在报告中可点击查看。

## 8. 配置与持久化

### 8.1 两套配置文件

| 配置文件 | 归属 | 用途 |
|---------|-----|------|
| `atprobe.yaml` | M5 CLI（M5 §3.5） | CLI 与引擎共享的参数默认值（端口、超时、压测阈值等） |
| `ui-presets.yaml` | M6 独有 | GUI 状态：窗口布局、端口预设、快捷指令、历史指令、上次勾选 |

- M6 启动时**优先加载 `atprobe.yaml`**（与 CLI 一致的默认值），再叠加 `ui-presets.yaml` 的 GUI 个性化。
- 两套文件解耦：CLI 用户不受 GUI 预设影响，反之亦然。

### 8.2 GUI 状态持久化内容

- 窗口尺寸/位置、三栏比例、当前激活 Tab。
- 端口预设列表（§3.4）。
- 快捷指令按钮列表（§4.4）。
- 最近执行参数（参与端口、阈值等），便于下次快速复现。

## 9. 手动调试 vs 用例执行：模式对比

这是 M6 最需要厘清的设计点，两种模式的边界：

| 维度 | 手动调试模式 | 用例执行模式 |
|------|-------------|-------------|
| 数据路径 | 用户 → **M1** | 用例 YAML → **M3 引擎** → M1 |
| 是否走引擎 | ❌ 不走 | ✅ 走 |
| 响应判定 | 用户肉眼看 | 自动断言（M2 §4） |
| 变量提取 | ❌ 无 | ✅ 有（M2 §5） |
| 结果记录 | 仅原始日志（M1 §7） | ExecutionResult + 原始日志 |
| 报告 | ❌ 不生成 | ✅ 生成（M4） |
| 失败处理 | 无（手动决定下一步） | on_failure/retry/poll（M3 §4） |
| 端口占用 | 独占选中端口 | 引擎占用参与端口集 |
| 典型场景 | 未知指令探索、快速验证、调试 | 回归测试、压测、可复现验证 |

> 手动调试是「探索」，用例执行是「沉淀」。用户通常先用手动调试摸清指令行为，再固化为用例。

## 10. 技术约束与非功能需求

### 10.1 跨平台与 UI 技术约束

- PRD §10.1 约束运行环境为 Windows / Linux / macOS。
- 技术选型需支持三平台桌面 GUI（具体框架在 TSD-M6 技术选型中定，候选如 Tauri/Electron/Qt/原生）。
- **选型须满足现代化 UI 能力**：支持 CSS 或等价样式系统（圆角/阴影/过渡动效）、矢量图标、浅/深主题、等宽与比例字体混排。纯原生控件若达不到 §2.2 视觉要求则需自绘或换框架。
- **选型须原生支持多文档选项卡**，或可低成本实现选项卡的增删/拖拽/多开/持久化（§2.3），这是扩展性架构的基石。

### 10.2 线程模型

- GUI 主线程（事件循环）与 M3 引擎执行线程分离，避免执行卡 UI。
- M1 串口 I/O 在独立线程（M1 已有异步收发能力）。
- 进度事件通过线程间消息机制从引擎线程投递到 GUI 线程刷新。

### 10.3 性能

- 长时间监控的端口数据流不能撑爆内存：响应区/监控区采用**环形缓冲 + 行数上限**（如保留最近 10000 行），超出自动丢弃旧行。
- 原始日志落盘由 M1 负责（§7），GUI 缓冲与落盘解耦。

### 10.4 数据量保护

- 手动调试多行发送、数据流发送大文件时，进度提示 + 可取消（委托 M1 §4.3 操作取消）。

### 10.5 UI 架构与扩展性

- **选项卡类型注册表**（§2.3）是扩展性的契约：每个功能 = 一个注册类型 + 内容视图实现。新增功能不改侧边栏/状态栏/选项卡栏这三个稳定外壳。
- **选项卡内容视图解耦**：各类型视图只依赖 M1-M5 的接口，相互独立，可独立加载/卸载（为未来插件化预留）。
- **GUI 状态与逻辑分离**：视图层只渲染数据、转发用户操作；所有业务逻辑在 M1-M5。视图可被替换（如未来换 UI 框架）而不动业务层。
- **设计令牌（design token）**：颜色、间距、字号、圆角、阴影等视觉参数集中定义为令牌，主题切换通过切换令牌集实现，不硬编码到各视图。
- **国际化预留**：UI 文案通过资源文件管理（第一阶段仅中文），为后续多语言预留，不硬编码字符串。
